title: Solution structure
description: Where entities, modules, and schema configuration belong relative to the module-boundary rule — keep entities in a shared-kernel namespace, treat configuration as platform (not feature-owned), and graduate to bounded contexts only when they earn their keep.
---

[Project structure](/docs/getting-started/project-structure) covers the mechanical layout the
generators rely on. This page is the **decision behind that layout**: given that Elarion's
module-boundary analyzer ([`ELMOD002`](/docs/concepts/cross-module-communication)) treats everything
inside an `[AppModule]` as module-internal, where do shared things like entities go?

## The rule that drives everything

The boundary analyzer treats every type declared inside a module as **module-internal** — reachable
from another module only through a [`[ModuleContract]`](/docs/concepts/cross-module-communication)
interface. Declaring `DependsOn` does **not** grant access to another module's internals; it only
orders initialization.

Concretely, a module's internal **code** — a `[Service]`, a handler, or an `[EntityConfiguration]` —
declared *inside* module A cannot be referenced from module B. Types declared under **no** `[AppModule]`
(the shared kernel) are never flagged, so every module may depend on them freely. **Entities are
deliberately part of that shared kernel** — they are shared data, not module-internal code.

That distinction — code is module-internal, data is shared — decides where entities go.

Said as a model: **a feature is a *plugin* over a *platform* data model.** The platform is the shared
substrate — the entities, their configuration, and the framework. A feature composes it into behavior
and may touch *any* entity, but it never depends on another feature except through a
[`[ModuleContract]`](/docs/concepts/cross-module-communication). The dependency arrow points
**feature → platform**, never the reverse and never feature → feature. Touching many *entities* is
using the platform; depending on another *feature* is the thing to avoid.
[ADR-0007](https://github.com/swimmesberger/Elarion/blob/main/docs/decisions/0007-data-is-platform-module-as-plugin.md)
records this model and what it implies for where configuration belongs.

## Entities are shared kernel, not module-internal

**Do not put entities inside a module.** Domain entities cross-reference each other through foreign-key
navigation properties, and handlers routinely query across aggregates — so a single module cannot own
them, and a `[ModuleContract]` interface cannot model an EF relationship. Modules are *feature*
separation, not *data* separation: every module reaches the whole database through the shared
`IAppDbContext` by design (real data isolation would be a separate `DbContext`). The boundary analyzer
reflects this — it flags a module's internal *code* (`[Service]`, handler, `[EntityConfiguration]`) but
**never an entity** — so a cross-aggregate reference is never flagged. (Reference other aggregates by id
rather than by navigation property; see [the delete-the-module test](#the-delete-the-module-test) for why.)
Keeping entities in one shared-kernel namespace is therefore a design convention that gives shared data an
obvious home, not an analyzer workaround.

Keep entities and their enums in a **shared-kernel namespace under no `[AppModule]`** — for example
`MyApp.Application.Domain` — that every module may depend on and that depends on no module.

```text
src/
  MyApp.Application/
    Domain/                  ← shared kernel: entities + enums (NOT in any [AppModule])
      Client.cs              (plain entity — no marker)
      Invoice.cs             (plain entity; references Client by id, not a navigation)
      InvoiceStatus.cs       (enum)
    Persistence/             ← platform: entity configuration = the schema *definition* (under no [AppModule])
      ClientConfiguration.cs   [EntityConfiguration] : IEntityTypeConfiguration<Client>
      InvoiceConfiguration.cs  [EntityConfiguration] : IEntityTypeConfiguration<Invoice>
    Modules/
      Clients/               ← vertical slice: handlers, validators, services — behavior only
        ClientsModule.cs       [AppModule("Clients")]
        Handlers/ …
      Invoicing/
        InvoicingModule.cs     [AppModule("Invoicing")]
        Handlers/ …
  MyApp.Infrastructure/      ← platform: concrete DbContext, provider, migrations (schema *provisioning*)
  MyApp.Api/                 ← host: [GenerateModuleBootstrapper], middleware, transport wiring
```

## Where configuration lives

The EF Core generator discovers `[EntityConfiguration]` classes **wherever they live** — the attribute is
the single source of truth, driving both the generated `DbSet<T>` and the configuration application, and it
emits direct `ApplyConfiguration<T>` calls into a generated `ConfigureEntities(ModelBuilder)` on the
context. Placement is therefore a **convention, not a generator requirement**.

Because a configuration describes the one shared model every feature draws on, it is *platform*, not
feature behavior. The recommended home is a shared `Persistence` layer — a sibling of `Domain`, under no
`[AppModule]` — kept as a flat folder while there is a single `DbContext` (one folder per `DbContext` once
you use scopes). The `samples/Billing` app currently co-locates each `[EntityConfiguration]` with its
module — a simpler form the generator also accepts; the recommended layout is the shared `Persistence`
layer. See
[ADR-0007](https://github.com/swimmesberger/Elarion/blob/main/docs/decisions/0007-data-is-platform-module-as-plugin.md)
for the platform/plugin rationale and why configuration is not feature-owned.

Wherever it sits, a configuration is safe across the module boundary for two independent reasons:

1. It references only the shared-kernel entity (`Client`), never another module's internals.
2. `ELMOD002` inspects only the **dependency surface** — constructor parameters, fields, and
   properties — never a method body like `Configure`.

The per-assembly configuration manifest pairs the discovered `[EntityConfiguration]` configs with the
shared entities across assemblies, and the generated `ConfigureEntities` lives in the **context's** assembly
(which carries no module analyzer). Infrastructure instantiates nothing per-entity — it just calls
`ConfigureEntities(modelBuilder)` from `OnModelCreating`.

<Callout type="info">
  Both layers are *platform*, at different tiers. `Persistence` holds the schema **definition** — the
  `[EntityConfiguration]` classes, i.e. *what shape a `Client` table has* — while `Infrastructure` holds
  the schema **provisioning**: the concrete `DbContext`, provider, connection, interceptors, and
  migrations. Neither is module-owned — configuration is platform, not a feature concern.
</Callout>

## A separate `Domain` project is usually unnecessary

For **one bounded context / one `DbContext`**, a shared-kernel *namespace* inside the application project
is enough. A separate assembly only earns its keep when **multiple host projects share the code**. Until
then, `MyApp.Application.Domain` (a namespace, not an assembly) avoids a project boundary you would only
have to wire and reference.

<Callout type="warn">
  Whatever assembly declares `[EntityConfiguration]`/`[GenerateDbSets]` types must reference
  **`Elarion.EntityFrameworkCore`** (whose bundled generator emits that assembly's configuration manifest),
  not merely depend on a project that does. NuGet analyzer assets are not transitive. If you *do* split
  entities and their configurations into their own project, that project needs the reference directly —
  otherwise no manifest is emitted, the context resolves zero configurations, and no `DbSet`s are generated,
  with no error. See
  [Entity Framework Core → Setup](/docs/capabilities/entity-framework).
</Callout>

## The delete-the-module test

A module is meant to be a plugin: **delete its folder and every other module still compiles, untouched.**
That holds because cross-module *code* goes through `[ModuleContract]`s (`ELMOD002` keeps it honest) and
`[EntityConfiguration]` is discovered structurally, never referenced by name — so removing a feature
removes its behavior cleanly. Two disciplines keep the test green:

- **Reference other aggregates by ID, not by navigation property.** `Invoice.ClientId` (a `Guid`) leaves
  the model untangled, so removing the `Clients` feature does not ripple into `Invoicing`. A cross-module
  navigation creates an EF relationship that couples the two.
- **Treat new data as platform.** A feature that adds an entity, index, or column *extends the platform*;
  that data lives in the shared data layer and does **not** vanish when the feature folder is deleted — its
  removal is a deliberate migration step, because data is durable and shared. (How a plugin should own new
  typed data while staying cleanly removable is an open question —
  [ADR-0008](https://github.com/swimmesberger/Elarion/blob/main/docs/decisions/0008-bounded-contexts-and-the-graduation-path.md).)

## Graduating to a bounded context

The shared-kernel default is right until an aggregate cluster becomes a **genuinely separate bounded
context with its own `DbContext` and schema** — real *data* separation, not just feature separation. At
that point its entities become context-private (correctly, since nothing outside the context references
them). The graduation is incremental, not a rewrite:

| Stage | Layout |
| --- | --- |
| One bounded context, one `DbContext` | Shared-kernel **namespace** + a shared `Persistence` layer in the application project |
| Partition into separate `DbContext`s without splitting assemblies | **Scopes** — `[GenerateDbSets("ctx")]` / `[EntityConfiguration("ctx")]`, one `Persistence` folder per scope |
| An aggregate becomes its own bounded context | A self-contained **`Context.Application` / `.Infrastructure` / `.Contract`** assembly group owning its entities, configs, and migrations |

The by-ID reference discipline above is what keeps each promotion cheap. See
[ADR-0008](https://github.com/swimmesberger/Elarion/blob/main/docs/decisions/0008-bounded-contexts-and-the-graduation-path.md)
for the full bounded-context model — two boundaries / two enforcers, cross-context collaboration via
`[ModuleContract]` + the outbox, and the staged graduation path.
