---
title: Actors
description: In-memory state machines as plain classes — [Actor] gives you keyed, mailbox-protected instances with source-generated typed facades, Orleans-style single-threaded execution, and idle passivation. Single-node by design.
---

Handlers are stateless: a fresh scope per call, state in the database. Some problems are the opposite
shape — a **state machine per order, session, or device** whose in-memory state must survive across calls
and be mutated **sequentially**. Hand-rolling that means a `Channel` plus a reader loop, or locks over a
singleton, with the actual state machine buried under concurrency plumbing.

`Elarion.Actors` makes that shape declarative: a plain class marked `[Actor]` becomes a keyed,
mailbox-protected instance, and a source-generated **typed facade** is how everyone calls it. No base
class, no message types, no `Receive` method, no locks — the mailbox guarantees one message at a time, so
methods mutate fields like single-threaded code.

```csharp
[Actor]
public sealed class OrderFulfillmentActor(IActorContext<Guid> context, IEmailSender email)
    : IActorLifecycle {
    private FulfillmentState _state = FulfillmentState.Draft;   // plain fields, never locked

    public ValueTask OnActivateAsync(CancellationToken ct) => LoadAsync(context.Key, ct);

    public async Task<Result<Unit>> Ship(ShipmentInfo info, CancellationToken ct) {
        if (_state != FulfillmentState.Packed) {
            return AppError.Conflict($"Cannot ship in state {_state}.");
        }

        await email.SendShippedAsync(context.Key, ct);
        _state = FulfillmentState.Shipped;
        return Result.Success();
    }

    public Task<FulfillmentState> CurrentState() => Task.FromResult(_state);
}
```

The generator emits `IOrderFulfillment` (the class name minus the `Actor` suffix, prefixed `I`) mirroring
the public async methods. Callers resolve it through `IActorSystem`:

```csharp
public sealed class ShipOrderHandler(IActorSystem actors)
    : IHandler<ShipOrder, Result<Unit>> {
    public async ValueTask<Result<Unit>> HandleAsync(ShipOrder request, CancellationToken ct) {
        var order = actors.Get<IOrderFulfillment>(request.OrderId);
        return await order.Ship(request.Info, ct);
    }
}
```

A facade is a cheap, always-valid **address**, not a live handle: `Get` never activates anything, and the
facade stays valid across passivation. A rename of an actor method is a compile error at every call site.

### What the generated facade looks like

For the `OrderFulfillmentActor` above, the generator emits (alongside an internal implementation and
per-method work items you never touch):

```csharp
// <auto-generated/> — public, next to the actor, in the actor's namespace
public interface IOrderFulfillment : IActorFacade<Guid> {
    Task<Result<Unit>> Ship(ShipmentInfo info, CancellationToken cancellationToken = default);
    Task<FulfillmentState> CurrentState(CancellationToken cancellationToken = default);
}
```

The shape rules are mechanical:

- **Name**: actor class name minus a trailing `Actor`, prefixed `I` (`OrderFulfillmentActor` →
  `IOrderFulfillment`; override the base name with `[Actor(Name = ...)]`).
- **Marker**: the interface extends `IActorFacade<TKey>` for a keyed actor (the key type comes from the
  `IActorContext<TKey>` constructor parameter) or the non-generic `IActorFacade` for a singleton — this
  is what makes `IActorSystem.Get` key-type-safe at compile time.
- **Methods**: every public async method is mirrored with its exact return shape
  (`Task`/`Task<T>`/`ValueTask`/`ValueTask<T>`) and parameters. If the actor method has no
  `CancellationToken` parameter, the facade appends a trailing optional one anyway — it governs the
  caller side of the call (queue wait and call timeout) even when the actor body never observes it.
  Lifecycle methods (`OnActivateAsync`/`OnDeactivateAsync`) stay off the facade.
- **No wrapping**: return types are passed through untouched — a `Result<T>`-returning actor method
  gives a `Result<T>`-returning facade method, and exceptions rethrow unwrapped.

## Keyed and singleton actors

Declaring a constructor parameter `IActorContext<TKey>` makes an actor **keyed** (virtual, Orleans grain
style): one activation per distinct key, created lazily by the first message, passivated after an idle
timeout (default 5 minutes — state is dropped, the next message re-activates). `string`, `Guid`, `long`,
and `int` keys have dedicated `Get` overloads; any other `notnull` key works via `GetByKey<TFacade, TKey>`.

Without a key declaration the actor is a **process singleton**, resolved with `Get<TFacade>()`.

Optional lifecycle hooks come from `IActorLifecycle`: `OnActivateAsync` runs before the first message
(load state), `OnDeactivateAsync` after the mailbox drains on passivation or shutdown (flush state). Both
run under the single-threaded guarantee. In-memory actor state is deliberately ephemeral — anything that
must survive passivation belongs either in [snapshot-backed `IActorState<TState>`](#durable-state-snapshots)
(the declarative route) or in storage the actor loads and flushes through these hooks by hand.

Every activation owns a **DI scope**: constructor parameters beyond the context resolve from it via
generated `GetRequiredService` calls, and the scope is disposed on passivation.

## Durable state (snapshots)

Declaring a constructor parameter `IActorState<TState>` gives the activation **snapshot-persisted
state** (ADR-0047) — the Orleans `IPersistentState<T>` shape:

```csharp
// The state record is the QUERY CONTRACT: interpretation and pure transitions live on it, so
// everything that reads the snapshot — the actor on its home, an IActorStateReader query on any
// other instance, ops with SQL over the jsonb column — shares one set of rules.
public sealed record FulfillmentState {
    public required string Stage { get; init; }
    public int Attempts { get; init; }

    public static FulfillmentState Initial { get; } = new() { Stage = "new" };

    public bool IsStuck => Attempts >= 3 && Stage != "shipped";            // derived interpretation

    public FulfillmentState Shipped() => this with { Stage = "shipped" }; // pure transition
}

[Actor]
public sealed class OrderFulfillmentActor(
    IActorContext<Guid> context,
    IActorState<FulfillmentState> state) {

    public async Task Ship(ShippingInfo info, CancellationToken ct) {
        state.State = (state.State ?? FulfillmentState.Initial).Shipped();
        await state.WriteStateAsync(ct);     // the explicit durability point — the commit
        // side effects that must not repeat go here, AFTER the successful write
    }
}
```

The member surface deliberately mirrors Orleans' `IPersistentState<TState>`
(`State`/`RecordExists`/`Etag`/`ReadStateAsync`/`WriteStateAsync`/`ClearStateAsync`), so when an app
outgrows the single-node runtime, every state call site compiles against Orleans unchanged — migrating
an actor means swapping the constructor parameter type, nothing inside the methods. The deltas are
call-site-compatible improvements: `ValueTask` returns, optional `CancellationToken`s, and a nullable
`State` (no parameterless-constructor requirement, so `required`-property records work).

- **Loaded before the first turn.** The runtime reads the latest snapshot after construction and before
  `OnActivateAsync`, so the hook and every message observe durable state; a passivated actor re-reads its
  snapshot on the next activation. A load failure fails the activation loudly — an actor never runs
  against half-loaded state. (`ReadStateAsync` is that same read, exposed for deliberate refreshes.)
- **Written only when you say so.** `WriteStateAsync` persists the whole snapshot; `ClearStateAsync`
  deletes it. Passivation never flushes implicitly — write after every mutation that must survive, and
  batch several mutations into one write when the actor is hot. `State` is `null` until first assigned
  when no snapshot exists (`RecordExists` tells you which case you are in); the actor constructs the
  initial value.
- **Conflicts never lose updates — and the runtime absorbs them once.** Every write is ETag-guarded: a
  snapshot changing underneath an activation (another process hosting the same key, or an out-of-band
  edit) makes the write conflict instead of silently clobbering. The conflicted turn is not surfaced to
  its caller: the stale activation **passivates and the whole turn re-runs once on a fresh activation**
  that loaded the winning snapshot, under the same call timeout. Only a second consecutive conflict —
  live contention, i.e. sustained double-hosting — reaches the caller as
  `ActorSnapshotConcurrencyException`; every conflict also logs a warning and increments the
  `actor.snapshot.conflicts` counter, so the misconfiguration stays visible. Two consequences: a
  conflicted turn executes **twice** (write your turn as a reapplyable function of the loaded state),
  and side effects before the successful write are at-least-once on the conflict path — side effects
  that must not repeat belong after the write, or must be idempotent.
- **Canonical JSON.** Snapshots serialize through Elarion's canonical serialization, so `TState` must be
  registered in a source-generated `JsonSerializerContext` that participates in the canonical options
  (like any other module type). Under reflection-off serialization a missing registration fails the
  activation with a clear error.
- **Not transactional with business writes.** A snapshot write is its own commit. If a piece of state
  must change atomically with business data, it belongs in regular entities written by a handler inside
  its transaction — the actor is then a coordinator over that data, not its store.

### Designing applications around snapshot state

Snapshot state is a **published-state model**. These rules are not style suggestions — an application
that ignores them gets queries that disagree with the actor, side effects that duplicate, or state
that stops being readable off-home:

1. **The state record is the query contract — interpretation lives on the record, never in actor
   methods.** Constants (thresholds), derived flags (computed properties), and pure transitions
   (methods returning the next state) belong on `TState`. The record is a shared type: the actor
   deserializes it on its home, `IActorStateReader` deserializes it on any other instance, ops reads
   it with SQL — logic *on the record* runs identically everywhere; logic in an actor method runs
   only where the actor runs. The test: if a facade query and a reader-based query could ever
   disagree about what the state *means*, the interpretation is in the wrong place.
2. **Actor methods apply transitions, write, and perform side effects — nothing else.** The canonical
   turn is three steps: `state.State = (state.State ?? TState.Initial).SomeTransition(…)`, then
   `WriteStateAsync`, then side effects. Pure, reapplyable transitions are also exactly what conflict
   recovery requires: a conflicted turn passivates and re-runs against the reloaded snapshot, and only
   a mutation expressed as a function of the loaded state survives that correctly.
3. **`WriteStateAsync` is publication.** Unwritten in-memory state is uncommitted — invisible to
   readers and other instances, lost on a crash, by design. Decide what observers must be able to see
   and write at those points. Side effects that must not repeat go *after* the successful write: the
   write is the commit point, and a turn whose write conflicts re-runs once.
4. **Queries read snapshots; commands go through the actor.** `IActorStateReader.ReadAsync<TState>`
   (composed with regular entities in an ordinary query handler where needed) runs on any instance and
   returns published truth carrying the record's full interpretation — rule 1 is what makes this
   lossless. Facade calls, including facade *queries*, run only where the actor runs. A query that
   must observe uncommitted state or be ordered against in-flight turns is home-only by construction —
   and usually a sign the actor should publish (write) more eagerly.
5. **Evolve the record shape-tolerantly instead of migrating in `OnActivateAsync`.** Activation-time
   migration fixes only the home's view; readers and SQL see raw payloads. Add new properties as
   optional/defaulted (nullable with a fallback in the computed interpretation), keep existing wire
   names stable, and treat a breaking shape change as what it is — a new contract that needs a
   backfill of the snapshot rows.
6. **Choose the write cadence deliberately — it decides what queries mean.** There are three cadences,
   and they must not be mixed up:
   - **Write-through** (write after every mutation — the coordinator shape): the snapshot always
     reflects the last completed turn, so `IActorStateReader` is as fresh as any database query
     (read-committed semantics — an in-flight turn hasn't published yet, same as an uncommitted
     transaction). This is the only cadence where the reader is a first-class query path. It is also
     the cadence closest to "just a table with a version column" — apply the shape-3 honesty test
     before choosing it.
   - **Checkpointing** (mostly in-memory, periodic or milestone writes): the snapshot is a
     **warm-restart checkpoint, not a query mechanism**. The reader returns bounded-stale data (up to
     one checkpoint interval old) — fine for after-the-fact reporting, wrong for anything presented as
     current. Use it to make hot shape-2 state survive restarts cheaply; never point a "live" view at
     it.
   - **No snapshot** (pure hot/ephemeral state): the reader has no role — there is nothing durable to
     read.
   **Real-time observation of hot in-memory state is a push problem, not a pull problem.** No node can
   query another node's memory (there is deliberately no forwarding), and polling the reader against a
   checkpoint is a staleness bug waiting to be filed. The actor — on its home — publishes
   [client events](/docs/capabilities/events/client-events) (throttled, small current-value payloads);
   the PostgreSQL fan-out delivers them to **every** instance's connected browsers, so observers get
   real-time data without knowing or caring where the actor lives. A *service* (not a browser) that
   needs pull-based real-time reads of live actor memory from arbitrary nodes has crossed the
   placement/forwarding line — that is the Orleans trigger, not a reason to stretch the reader.

The Billing sample is the reference implementation of both halves: `ClientDunningState` carries
`EscalationThreshold`, the `NeedsAttention` interpretation, the `RecordOverdue()` transition, and its
own `SnapshotKey` (so actor and queries can never disagree on identity); the actor's `[ConsumeEvent]`
method is apply → write → log-after-write; and the `GetClientDunning` query handler is the read path —
`IActorStateReader` plus the record's interpretation, no `IActorSystem`, runnable on any instance.

The store behind the seam is `IActorSnapshotStore`. The shipped implementation is
`Elarion.Actors.PostgreSql`: one `elarion_actor_snapshots` table on the app's existing Postgres, `jsonb`
payload (actor state stays inspectable with plain SQL), version-guarded change-tracker-free writes.
Wire it with the marker on your `[GenerateDbSets]` context and one registration:

```csharp
[GenerateDbSets]
[GenerateElarionActorSnapshots]
public partial class AppDbContext(DbContextOptions<AppDbContext> options) : DbContext(options);

services.AddElarionPostgreSqlActorSnapshots<AppDbContext>();
```

(Hand-written contexts call `modelBuilder.UseElarionActorSnapshots()` instead of the marker.) A different
backend implements `IActorSnapshotStore` and replaces the registration — the actor code never changes.

## Running on multiple instances

There is no placement and no forwarding: `IActorSystem.Get<T>(key)` always resolves an activation **on
the calling instance**. Five instances calling the same key means up to five independent activations —
each with its own mailbox; the single-threaded guarantee holds per instance. What snapshot state adds is
one shared row: the ETag becomes the *cross-instance* serialization point (whoever writes first wins;
every other activation's turn passivates its instance's activation and transparently re-runs once on the
reloaded snapshot). Cross-instance, the guarantee is therefore optimistic concurrency on writes, not
one-turn-at-a-time — a conflicted turn executes twice, so side effects before the write are
at-least-once. Pick a topology per actor:

1. **Per-instance actors** (caches, per-node rate limiters, batchers — usually no `IActorState`):
   independent activations per instance are the point. Nothing to do.
2. **Single-homed actors** (per-entity coordinators like an escalation latch) — **the recommended
   shape**: declare `[Actor(SingleHomed = true)]` and let the **actor home lease** (ADR-0048) elect
   exactly one instance as the home. See [Single-homed actors](#single-homed-actors-the-home-lease)
   below.
3. **Multi-writer everywhere** (no topology work): keep mutations reapplyable and side effects
   idempotent (or after the write), and let the runtime's conflict handling converge — a losing turn
   re-runs once on the reloaded snapshot; only sustained contention surfaces errors. Fine for
   low-contention keys; a hot shared key or true placement/forwarding is the signal to move to
   Orleans (its ADO.NET clustering runs membership on the PostgreSQL you already have — and the
   facade/`IActorState` surfaces are deliberately shaped so that migration is mechanical).

## Single-homed actors: the home lease

`[Actor(SingleHomed = true)]` declares that an actor must only run on one instance app-wide. The
enforcement is the **actor home** — a [role lease](/docs/decisions) (ADR-0049): one heartbeat-renewed
row on the app's PostgreSQL that elects exactly one instance as the holder of the `"actors"` role (the
row is the whole membership protocol; failover is expiry, bounded by `LeaseDuration`, immediate on
graceful shutdown):

```csharp
[GenerateDbSets]
[GenerateElarionActorSnapshots]
[GenerateElarionRoleLeases]                   // the lease table (ELROLE001 without [GenerateDbSets])
public partial class AppDbContext(DbContextOptions<AppDbContext> options) : DbContext(options);

services.AddElarionPostgreSqlActorHome<AppDbContext>();   // "actors" role lease + heartbeat + the home binding
services.AddElarionOutbox<AppDbContext>(o =>              // event delivery follows the lease
    o.DeliveryGate = (sp, _) => ValueTask.FromResult(sp.GetRequiredService<IActorHomeLease>().IsHeld));
```

(The role lease is a general leader-election primitive — `Elarion.Coordination.PostgreSql`, keyed
`IRoleLease` per role — usable for any "exactly one instance runs this" background concern; the actor
home is its first consumer, bound via `AddElarionActorHome("actors")`.)

With those three lines a homogeneous deployment self-organizes: every instance publishes integration
events; the lease holder delivers them and hosts the single-homed actors (the generated `[ConsumeEvent]`
relays run where delivery runs); when the holder dies, another instance takes the lease and both move
together. Messages published while no instance is home simply wait in the outbox — nothing is lost.

The rules of the model:

- **Calls are gated, never forwarded.** A facade call to a single-homed actor on a non-home instance
  fails immediately with `ActorNotHomedException` naming the holder — a pointed error, not mystery
  latency. Feed single-homed actors through integration events (they follow the lease); a scheduled job
  should publish an event rather than call the facade (job claims elect a node independently of the
  home). Transparent forwarding is deliberately out of scope — needing it is the signal to move to
  Orleans.
- **Reads run anywhere — because interpretation lives on the record.** `IActorStateReader.ReadAsync<TState>(key)`
  returns the latest snapshot without activating anything — durable truth on any instance, carrying the
  state record's full logic (rule 1 of [designing applications around snapshot
  state](#designing-applications-around-snapshot-state); off-home reads lose nothing precisely because
  actor *methods* hold no interpretation). How fresh that is equals the actor's **write cadence**
  (rule 6): write-through → as fresh as any database query; checkpointing → bounded-stale, never a
  "live" view. Real-time observation of the home's in-memory state is push — client events fanned out
  to every instance — not reader polling. Commands go through the actor; queries read snapshots (or
  regular entities).
- **Unenforced without a lease.** With no `IActorHomeLease` registered (local dev, a one-instance
  deployment), `SingleHomed` is a declaration only — the actor system logs one warning per such actor so
  a multi-instance misconfiguration is visible.
- **Failover is honest.** The local `IsHeld` turns false a safety margin *before* the stored lease
  expires, so the old home stops before a new one can start; pathological overlaps (GC pause, clock
  skew) degrade to snapshot conflicts, which the ADR-0047 transparent retry absorbs losslessly. A
  database outage means "nobody is home" — fail-closed, never two homes.

The static alternative — a fixed web/worker split via module gating (`Modules:{Name}:Enabled=false` on
web instances) plus `RunDeliveryWorker = false` on publisher-only nodes — remains valid when you prefer
topology in configuration over self-election; the lease is for fleets of identical instances.

## Execution model: single-threaded, non-reentrant by default

One message runs start-to-finish before the next; an `await` inside a method holds the mailbox. Every
method body is a critical section — the model that feels like normal C#.

The corollary is the classic virtual-actor deadlock: if actor A awaits actor B while B awaits A, neither
mailbox drains. Every facade call therefore carries a **call timeout** (default 30 seconds) that fails the
call with a `TimeoutException` naming the actor and method — a cycle surfaces as a diagnosable error, not
a hang. Treat actor→actor request cycles as a design smell; prefer events or short, acyclic calls.

For actors that must stay responsive while a message awaits I/O — or that legitimately sit in a call
cycle — opt into Orleans-style **turn-based interleaving** with `[Reentrant]`:

```csharp
[Actor]
[Reentrant]
public sealed class DeviceSessionActor { ... }
```

A reentrant activation may start another message while one is awaiting, but turns never run in parallel —
every segment executes on an exclusive scheduler, so state is still never mutated concurrently. What you
give up is the "whole method is a critical section" guarantee: state observed across an `await` may have
been changed by an interleaved message. One caveat: `ConfigureAwait(false)` written **in the actor's own
methods** escapes the exclusive scheduler for the rest of that method and forfeits the guarantee — don't
use it in code that touches actor state (public methods and private helpers alike — flagged by
`ELACT006`). Libraries you call
may freely use `ConfigureAwait(false)` internally: context capture is per-method, so your method still
resumes on the actor's scheduler regardless of what the callee did. What *does* escape is a
state-mutating delegate handed to a library (`library.Subscribe(x => _state = x)` runs wherever the
library invokes it). There is deliberately no
`ReenterAfter`-style callback API: interleaving is declared once at the class, and method bodies stay
plain `async`/`await` (ADR-0042).

## Mailbox, cancellation, shutdown

Mailboxes are unbounded by default. `[Actor(MailboxCapacity = 1000)]` bounds one; a full mailbox either
makes callers wait asynchronously (`ActorMailboxFullMode.Wait`, default) or fails fast with
`ActorMailboxFullException` (`Fail`). There are no silent drop modes — every facade call is request/reply,
so a dropped call would strand its awaiting caller.

The `CancellationToken` a caller passes cancels a queued call before it runs and flows into the executing
method, linked with the call timeout and the activation's stopping token (`IActorContext.Stopping`). On
host shutdown the runtime drains every mailbox, cancels in-flight work once the host's shutdown deadline
fires, and runs `OnDeactivateAsync` per live activation.

## Configuring an actor

Most actors need no options — `[Actor]` with the defaults is the common case. Each knob exists for a
specific pressure; reach for one only when you have the matching problem.

| Option | Default | Reach for it when |
| --- | --- | --- |
| `Name` | class name minus `Actor` | You want a telemetry/facade name decoupled from the class name (rename the class without moving dashboards), or two actors would otherwise derive the same `I{Name}` facade. |
| `KeyType` | inferred from `IActorContext<TKey>` | Rarely — only to make an actor keyed when its constructor does **not** take the context (it never needs the key inside the body). Declaring the context parameter is the idiomatic route; if both are present they must agree (`ELACT004`). |
| `MailboxCapacity` | `0` (unbounded) | A single key could accumulate a backlog faster than the actor drains it and memory is a real risk; a bound caps in-flight work per activation. |
| `MailboxFullMode` | `Wait` | The mailbox is bounded and you need load-shedding rather than backpressure (see below). |
| `IdleTimeoutSeconds` | `0` (→ 5 min) | You want to trade the default memory-reclamation behaviour for something else (see below). |
| `CallTimeoutSeconds` | `0` (→ 30 s) | A single call is legitimately long, or you must disable the deadlock backstop (see below). |

The three that need real judgment:

**`MailboxFullMode` — backpressure vs. load-shedding.** On a bounded mailbox, `Wait` (default) makes a
caller await space, so a slow actor naturally slows its callers — the right default, and the only safe
choice when the caller has no fallback. `Fail` rejects immediately with `ActorMailboxFullException`;
pick it when a queued call would be worthless by the time it ran (an interactive request with a tight
deadline, or a path with a fallback) and shedding load beats letting the queue grow.

**`IdleTimeoutSeconds` — passivation.** The 5-minute default suits **high-cardinality keyed actors**
(per-order, per-session, per-device) where most keys go cold and their memory + DI scope should be
reclaimed; the next message re-activates and reloads state via `OnActivateAsync`. Set **`-1` (never
passivate)** for a singleton or a small fixed set of long-lived actors — a coordinator, a bounded worker
pool — where re-activation churn costs more than staying resident, or where in-memory state is expensive
to rebuild and nothing backs it in storage. Shorten it under memory pressure when state is cheap to
reload; lengthen it when reload is expensive and traffic is bursty. Remember passivation **drops
in-memory state**: if it must survive, keep it in [`IActorState<TState>`](#durable-state-snapshots) (and
`WriteStateAsync` after mutations) — otherwise accept the loss for cache-like actors.

**`CallTimeoutSeconds` — the deadlock backstop.** 30 seconds catches the actor→actor cycle (A awaits B
awaits A) that would otherwise hang a mailbox forever. Raise it only for an actor with a genuinely long
single call — better, move long work out of the message. Set **`-1` (no timeout)** only when you are
certain no call cycle exists **and** the caller supplies its own cancellation (e.g. a supervised
long-running loop message), because a real deadlock then never surfaces.

Concurrency is a separate, class-level decision — the
[`[Reentrant]`](#execution-model-single-threaded-non-reentrant-by-default) attribute, not an `[Actor]`
option.

## Wiring

Actors follow the standard module flow. Reference `Elarion.Actors` (runtime) and `Elarion` (the bundled
generator), and opt the assembly in:

```csharp
[assembly: GenerateActors]   // or the umbrella [assembly: UseElarion]
```

Under `[GenerateModuleBootstrapper]` nothing else is needed: each module's actors register through the
generated `Add{Module}Actors`, called inside the module's feature gate — a disabled module's actors
disappear like its handlers. Without the bootstrapper, call `services.Add{Module}Actors()` yourself (it
also wires the actor system). An actor outside every module warns (`ELACT003`) and is not registered.

Actors are module-internal like any implementation type: expose one to another module only through a
[`[ModuleContract]`](cross-module-communication.mdx). They intentionally have **no decorator pipeline** —
no `[RequirePermission]`, `[FeatureGate]`, or validation. An actor is internal machinery invoked by an
already-gated handler; the handler remains the transport-facing gate.

## Actors and events

Feeding integration events into an actor is common — put `[ConsumeEvent]` directly on an actor method
and the generator emits the relay that routes the event into the actor's mailbox:

```csharp
[Actor]
public sealed class OrderFulfillmentActor(IActorContext<Guid> context) {
    // OrderShipped carries exactly one Guid → the key is inferred. The method is public (the relay
    // reaches it through the facade), so it is also directly callable — an honest, dual-purpose op.
    [ConsumeEvent]
    public async Task OnShipped(OrderShipped e) {
        // ordinary actor code — mutate state, call siblings via `this`, never via the facade.
    }
}
```

The method's single event-typed parameter is the consumed event. Because the emitted relay is a real
handler-form integration consumer, it composes the guarantees in the right order — the three things a
hand-written relay used to carry are now generated:

- **Dedupe before the mailbox.** The relay gets the default-on [inbox](idempotency.mdx) (keyed by
  consumer identity + `MessageId`), so at-least-once outbox redelivery is swallowed *before* anything
  reaches the actor. Its inbox decorator runs before its body enqueues to the mailbox, so ordering
  (inbox → mailbox) holds by construction.
- **Key extraction.** The actor's key type is known at generation time. When the event has exactly one
  property assignable to it, the key is inferred; otherwise name the property with
  `[ActorKey(nameof(OrderShipped.OrderId))]`. A singleton actor needs no key. When the generator can
  neither infer nor read a valid `[ActorKey]`, it reports `ELACT008`.
- **The plane boundary.** Only **integration events** may reach an actor — a domain-event parameter is
  rejected with `ELACT010`. A domain-event consumer runs inside the emitting command's transaction and
  scope; an actor runs in its own scope on its own schedule, so awaiting an actor from a Plane A
  consumer abandons the same-transaction contract that plane exists for. To react to a domain change,
  publish an integration event and consume that (after commit), or have the command's handler call the
  actor directly after its own transaction completes.

Failure semantics compose correctly: if the actor call fails (or times out), the relay's `Result`
fails and the outbox retries the delivery — which the inbox then dedupes against the previous attempt.

**The consumed method is `public`.** The generated relay is exactly what you'd hand-write — it injects
`IActorSystem` and calls `actors.Get<IOrderFulfillment>(key).OnShipped(e)` through the public facade —
so the method must be `public` (and is thus on the facade). Non-public + `[ConsumeEvent]` is rejected
(`ELACT009`). Routing through the facade is deliberate: the generator only automates a call you could
type yourself, rather than unlocking a private enqueue path no hand-written relay could reproduce.

<Callout type="warn">
Inside any actor method, reuse a sibling method by calling it on `this` — a direct, in-turn local
call. Never route through the facade (`actors.Get<IOrderFulfillment>(key).Other()`): that re-enqueues
onto the actor's own mailbox and self-deadlocks a non-reentrant actor (the call-timeout backstop
fires).
</Callout>

A hand-written relay — a separate `[ConsumeEvent]` class implementing `IHandler<TEvent>` that calls
`actors.Get<…>(key).Method(…)` — remains valid for cases the attribute form can't express (fan-out to
several actors from one event, non-trivial key derivation).

## Observability

Register the `Elarion.Actors` ActivitySource and Meter with your OpenTelemetry providers. Each facade call
emits a caller-side `actor.call {Actor}.{Method}` span (kind `Client`) that parents the actor-side
`actor.process` span (kind `Internal`), so a trace crosses the mailbox like an RPC hop. Both spans carry
`elarion.actor`, `elarion.actor.method`, and `elarion.actor.key` — the key rides on **spans only**, never
on metrics, because span attributes tolerate high cardinality and metric tags don't. Exceptions are never
wrapped: the caller's `await` rethrows the actor-side exception with its original stack trace, the span
records it (`AddException` + error status, per OTel conventions), and nothing is retried behind your back.

Metrics (all tagged `elarion.actor`; message-level ones also `elarion.actor.method`, and the counter an
`elarion.actor.outcome` of `ok`/`error`/`canceled`/`timeout`/`abandoned`):

| Metric | Instrument | Meaning |
| --- | --- | --- |
| `actor.message.count` | counter | Processed messages by outcome (`timeout` = the deadlock backstop fired). |
| `actor.message.duration` | histogram (s) | Execution time, excluding queue wait. |
| `actor.message.queue_wait` | histogram (s) | Time from enqueue to execution start — rising values mean a hot mailbox. |
| `actor.mailbox.pending` | up-down counter | Messages enqueued or executing right now — the backpressure signal to alert on. |
| `actor.activations.active` | up-down counter | Live activations. |
| `actor.activations.count` | counter | Total activations (re-activation churn shows up here). |
| `actor.activations.failed` | counter | Activations whose constructor or `OnActivateAsync` threw (each failure also failed its queued calls). |
| `actor.snapshot.conflicts` | counter | Snapshot concurrency conflicts (ADR-0047). A steady rate means two processes host the same actor keys — alert on it. |

This deliberately matches the signal set actor operators expect from Proto.Actor's metrics or Akka.NET's
Phobos (mailbox depth, processing time, activation/failure counts, per-instance span identity). Durations
follow the OTel semantic conventions — **seconds**, with the standard bucket boundaries supplied as
instrument advice — like every Elarion duration histogram. The one knowing deviation: there are
**no dead-letter or restart metrics** because the model has neither — no message can be dropped (every
call is request/reply; rejected calls throw to their caller) and activations are never restarted behind
your back (a failed activation fails its calls; the next call re-activates).

## Routing and worker pools (the Akka router patterns)

Akka.NET ships router actors — round-robin pools, consistent-hash groups, broadcast — as dedicated
machinery. In the virtual-actor model those collapse into **keying**: the actor identity *is* the
routing function, so there is no router surface to configure. The classic patterns translate
directly:

```csharp
// "Consistent hashing" — address by identity. Every message for this device lands on the same
// activation, serialized. This is the pattern you want 95% of the time.
var twin = actors.Get<IDeviceTwin>(deviceId);

// Partition-affine worker pool — bound parallelism to K workers while keeping per-entity ordering:
// all work for one entity hashes to the same worker, different entities spread across the pool.
private const int PoolSize = 8;
private IImportWorker WorkerFor(Guid entityId) =>
    actors.Get<IImportWorker>((int)((uint)entityId.GetHashCode() % PoolSize));

// Round-robin pool — spread independent jobs across K workers when no per-entity ordering matters.
private int _next;
private IImportWorker NextWorker() =>
    actors.Get<IImportWorker>((int)((uint)Interlocked.Increment(ref _next) % PoolSize));

// "Broadcast" — fan a call out to a known key set.
await Task.WhenAll(shardIds.Select(shard => actors.Get<IShard>(shard).Refresh(ct).AsTask()));
```

Prefer the **partition-affine** form over round-robin: if ordering per entity didn't matter, the
work probably shouldn't be in an actor at all — purely stateless parallelism is a handler (or a
`[Service]`) plus `Task.WhenAll`/`Parallel.ForEachAsync`, where the scheduler can use every core
without mailbox serialization. A worker-pool actor earns its place when each partition needs
serialized processing or carries working state (batch buffers, connection handles, rate windows).

Two sizing notes: pool keys are ordinary actor keys, so idle workers passivate like any activation
(a fixed pool "re-materializes" on demand); and a bounded `MailboxCapacity` on the worker gives you
per-partition backpressure for free.

## When (not) to use an actor

**Default to the database.** In a classical web application, concurrency problems are database problems,
and the ordinary stack already solves them: two users editing the same record is
[optimistic concurrency](/docs/capabilities/entity-framework) (a version column); a webhook arriving
twice is [`[Idempotent]`](/docs/capabilities/idempotency) or the consumer inbox; "an order number must be
unique" is a constraint; "one export per tenant at a time" is a
[scheduled job](/docs/capabilities/scheduling) behind claims; reliable follow-up work is the
[outbox](/docs/capabilities/events/integration-events). A transaction plus a `WHERE version = @expected`
update *is* serialization — it just serializes at commit time instead of queueing callers. **Most
applications need zero actors**, and an actor is never the fix for "two requests raced on a row."

The honest test: **an actor is justified only when the unit of consistency is a live, in-memory thing —
something the database structurally cannot be.** That happens in exactly three shapes:

1. **The actor owns a stateful external resource.** A proprietary TCP session per machine, a device
   command channel, a per-tenant API client whose vendor allows one in-flight call (ordered calls, or a
   shared OAuth token that must refresh single-flight), an IMAP connection per mailbox. The serialization
   constraint lives *outside* the database — you cannot row-lock a socket, and holding a DB transaction
   open across an external call is the classic anti-pattern. One actor per resource *is* the connection
   owner: calls queue in its mailbox, the resource is touched by one turn at a time, and passivation
   closes it cleanly.
2. **The state is hot and ephemeral — the database would be pure overhead.** Live machine/telemetry data
   arriving many times per second from hundreds of devices, where dashboards read the current value and
   the data is worthless after a restart (stale readings mean nothing on startup — the device will send
   again); presence ("who is connected right now"); live progress of a long-running import pushed to
   browsers via [client events](/docs/capabilities/events/client-events); a write-behind buffer that
   coalesces many small events into one periodic DB flush. The defining property is **loss on restart is
   acceptable or naturally recovered** — which is why these actors usually need *no* snapshot state.
3. **A long-lived, event-driven decision process** — a coordinator that reacts to many integration events
   over time, holds a small decision state, and must act **exactly once** (the Billing sample's dunning
   escalation latch). This is the only shape that combines with
   [snapshot state](#durable-state-snapshots) (the decision must survive restarts) and often with
   [single-homing](#single-homed-actors-the-home-lease). Be honest here: the database *can* express this
   too (a state row + optimistic concurrency + the outbox for the side effect). The actor wins when the
   decision spans many events and the read-decide-act cycle would otherwise be a lock held across side
   effects or a retry loop in every consumer; if your version is one `UPDATE … WHERE version = @v`, use
   the table.

A junior-proof rule of thumb: **if you can sketch the solution as a table with a version column, do
that.** Reach for an actor only when the sketch keeps needing one of: a socket/client that must not be
shared, per-key state too hot or too transient to round-trip, or "react to a stream of events and act
once."

| Symptom | Reach for | Not |
| --- | --- | --- |
| Two users save the same record concurrently | Optimistic concurrency (version column) | Actor |
| A payment webhook may be delivered twice | `[Idempotent]` / the consumer inbox | Actor |
| Invariant: unique number, non-negative balance | DB constraint + transaction | Actor |
| Run one import per tenant at a time | Scheduled job (claims) / job queue | Actor |
| Reliable email after an order commits | Outbox integration event | Actor |
| Cache expensive query results | `[Cacheable]` / HybridCache | Actor |
| Fan out CPU/IO work over many items | Handler + `Task.WhenAll` | Actor |
| One TCP/API session per device or tenant, calls must be ordered | **Actor per resource** (shape 1) | Row locks |
| Live readings from hundreds of machines, dashboards read current values, restart resets | **Actor per machine** (shape 2, no snapshot) | A hot-loop `UPDATE` per reading |
| Presence / live progress pushed to browsers | **Actor** (shape 2) + client events | Polling a table |
| Real-time view of an actor's live in-memory state from any instance | **Client events pushed by the actor** (the Postgres fan-out reaches every instance's browsers) | Polling `IActorStateReader` (checkpoint staleness) |
| Escalate to collections exactly once after N overdue events | **Actor + snapshot** (shape 3) — or a state row if it stays one `UPDATE` | `SELECT FOR UPDATE` across an email send |

Everything else stays with the default building blocks: request/response work is a
[handler](handlers.mdx), shared logic is a `[Service]`, fan-out is an
[event](../capabilities/events/index.mdx), periodic work is a scheduled job. Stateless parallelism never
belongs in actors — that's handlers plus `Task.WhenAll`. For bounded, ordered parallelism see
[Routing and worker pools](#routing-and-worker-pools-the-akka-router-patterns).

**Actors are single-node.** In-memory activations on three nodes are three independent states — fine for
node-local concerns, wrong for cluster-wide authoritative state unless you make it so deliberately:
[snapshots](#durable-state-snapshots) make the state durable and conflict-safe,
[single-homing](#single-homed-actors-the-home-lease) makes one instance authoritative. What Elarion
deliberately does **not** ship is placement and call forwarding: if you need actors addressable from any
node with the runtime routing to them — the gaming/high-scale shapes — replace the runtime with a
dedicated engine (Orleans, Akka.NET, Proto.Actor) rather than growing this one, the same
replace-the-seam philosophy as the scheduler and cache defaults (ADR-0025, ADR-0042).

### The migration seam: how a facade maps to an Orleans grain

"Shape-compatible" is meant literally: the generated facade is the same *kind* of contract as an Orleans
grain interface — a key-addressed interface of `Task`-shaped methods resolved from a factory — so every
concept at the **call site** has a one-line Orleans equivalent:

| Elarion.Actors | Orleans |
| --- | --- |
| `public interface IOrderFulfillment : IActorFacade<Guid>` | `public interface IOrderFulfillment : IGrainWithGuidKey` |
| `actors.Get<IOrderFulfillment>(orderId)` (`IActorSystem`) | `grainFactory.GetGrain<IOrderFulfillment>(orderId)` |
| `Task`/`Task<T>` facade methods | grain interface methods (identical signatures) |
| `[Reentrant]` | `[Reentrant]` (same turn-based interleaving semantics) |
| `IActorLifecycle.OnActivateAsync` / `OnDeactivateAsync` | `Grain.OnActivateAsync` / `OnDeactivateAsync` |
| `IActorState<TState>` ctor param (`State`/`RecordExists`/`Etag`/`ReadStateAsync`/`WriteStateAsync`/`ClearStateAsync`) | `[PersistentState] IPersistentState<TState>` (same member names — state call sites compile unchanged) |
| Call timeout (default 30 s, `TimeoutException`) | response timeout (default 30 s) |
| Idle passivation (`IdleTimeoutSeconds`) | activation collection age |

Migrating a call site is therefore mechanical: swap the marker interface, resolve through
`IGrainFactory` instead of `IActorSystem`, and hand-write the interface the generator used to emit.
The honest print on what does **not** carry over automatically:

- **Message serializability is the real migration work.** In-process, arguments and results pass by
  reference — any type works, nothing is copied. Orleans crosses a wire (or at least a serializer):
  every parameter and return type must be serializable, and shared mutable objects stop being shared.
  This is a property of your DTOs, not of the call sites — which is exactly why keeping actor method
  signatures record/DTO-shaped from day one keeps the seam real.
- **The actor class body moves, the facade doesn't.** Orleans grains implement the interface directly
  (no generated facade layer) and take their key from the grain identity rather than an
  `IActorContext<TKey>` constructor parameter.
- **Exotic keys**: Orleans keys are `Guid`/`long`/`string` (plus compound forms); an actor using
  `GetByKey<TFacade, TKey>` with a custom key type needs a key mapping.
- **`ValueTask`-shaped methods**: `Task`/`Task<T>` port as-is; check your Orleans version's support
  before relying on `ValueTask` shapes in a facade you expect to migrate.

If migration is a realistic future for a given actor, the two cheap habits that keep the seam honest
are: serializable DTOs in the method signatures, and no reliance on by-reference argument sharing
between caller and actor.
