---
title: Packages
description: What each Elarion package contains, its expected dependencies, and when you need it.
---

Elarion is split into focused packages so you reference only what you use. Anything reusable moves
toward the `Elarion` family; application-specific domain concepts, database conventions, and
infrastructure stay in consuming projects.

## .NET packages

| Package | Purpose | Expected dependencies |
| --- | --- | --- |
| `Elarion.Abstractions` | Implementation-neutral attributes, handler contracts, result types, module metadata, scheduling contracts, the canonical JSON-serialization accessor (`IElarionJsonSerialization` / `ElarionJsonOptions` / `AddElarionJson` / `ConfigureElarionJson`), and source-generation triggers. | Application-level abstractions without provider-specific dependencies. |
| `Elarion` | Runtime primitives: decorators, modules, current-user access, the default authorizer, the in-memory scheduler, and the client-capability session bootstrap (`Elarion.Session` — `SessionHandler`/`AddElarionSession`, ADR-0030). The caching and resilience **seams and decorators** live here, but their implementations moved to opt-in packages (ADR-0017), so core depends only on `Microsoft.Extensions.*` abstractions. Bundles the Elarion source generator (handlers, services, modules, RPC/HTTP/MCP maps, validation resolvers, resilience policies, scheduled jobs, event consumers) as an analyzer. | `Elarion.Abstractions` and `Microsoft.Extensions.*` abstractions only; avoids ASP.NET Core, EF Core, and transport concerns. |
| `Elarion.Caching` | `HybridCache`-backed default `IHandlerCache` for handler result caching (`AddElarionHandlerCaching()`). **Required by `[Cacheable]`/`[CacheInvalidate]`** — the `HybridHandlerCache` was extracted out of `Elarion` core (ADR-0017). | `Elarion.Abstractions`; pulls `Microsoft.Extensions.Caching.Hybrid`. |
| `Elarion.Caching.PostgreSql` | The **recommended L2 distributed cache** for most apps: a PostgreSQL `UNLOGGED` table behind `HybridCache` (`AddElarionPostgreSqlHandlerCaching(connectionString)`), reusing your Postgres instead of operating Redis (`UseWAL = false` ⇒ an auto-created `UNLOGGED` table; ADR-0020). Redis stays the better L2 for very high-throughput, independently-scaled, or multi-region caches. | `Elarion.Caching`; pulls the official `Microsoft.Extensions.Caching.Postgres` (and Npgsql). |
| `Elarion.Resilience` | Microsoft/Polly-backed default `IResiliencePipelineRunner` (`AddElarionResilience()`). **Required by `[Resilient]` handlers and deferred scheduler retries** — the runner was extracted out of `Elarion` core, so `AddElarionScheduler` no longer auto-wires it (ADR-0017, breaking). | `Elarion`; pulls `Microsoft.Extensions.Resilience`. |
| `Elarion.Validation` | `Microsoft.Extensions.Validation`-backed default `IRequestValidator` for [declarative request validation](/docs/concepts/validation) (`AddElarionValidation()`): DataAnnotations on request DTOs, enforced by the framework `ValidationDecorator` and exported to every schema surface (ADR-0027). **Required to enforce validation attributes at runtime** — without it they are schema-documented but unenforced (`ELVAL002`). The `IRequestValidator` seam and the decorator stay in `Elarion.Abstractions`. | `Elarion.Abstractions`; pulls `Microsoft.Extensions.Validation` (runtime base only — Microsoft's bundled validation generator is excluded). |
| `Elarion.Scheduling.EntityFrameworkCore` | Cross-instance scheduler coordination over EF Core/PostgreSQL: per-occurrence claim rows make each recurring job occurrence execute on exactly one node (exact slots for cron, one-interval dedupe windows for fixed-rate/fixed-delay), plus a claim-retention purge worker — `AddElarionSchedulerEntityFrameworkCore<TDbContext>()`; `UseElarionSchedulerClaims(modelBuilder)` / `[GenerateElarionSchedulerClaims]` map the table. Bundles its generator. Targets PostgreSQL. | `Elarion.Abstractions` and EF Core relational APIs. |
| `Elarion.FeatureFlags.OpenFeature` | Default `IFeatureFlagService` over OpenFeature's `IFeatureClient` for `[FeatureGate]`/`[FeatureVariant]`; maps `ICurrentUser` into the OpenFeature `EvaluationContext` (targeting key + `UserId`/`Groups`) off-HTTP, so gating works with any OpenFeature provider (`AddElarionOpenFeature()`). | `Elarion.Abstractions` and `OpenFeature`; bring your own OpenFeature provider via `services.AddOpenFeature(...)`. |
| `Elarion.FeatureFlags.FeatureManagement` | Batteries-included, config-driven flags: wires the OpenFeature Microsoft.FeatureManagement provider behind the OpenFeature-backed `IFeatureFlagService` (`AddElarionFeatureManagement(configuration)`). | `Elarion.FeatureFlags.OpenFeature`, `OpenFeature.Hosting`, and `OpenFeature.Contrib.Provider.FeatureManagement` (a prerelease — NU5104 suppressed deliberately, ADR-0016). |
| `Elarion.Blobs` | Provider-neutral blob storage contracts and DTOs, plus the upload lifecycle (`BlobLifecycleState`, `IBlobLifecycle`). | No provider-specific storage dependencies. |
| `Elarion.Blobs.PostgreSql` | PostgreSQL-backed blob storage with EF Core model configuration and Npgsql content I/O, and the `IBlobLifecycle` implementation plus the `BlobGarbageCollector` sweeper — `AddElarionPostgreSqlBlobStore<TDbContext>()`; `UseElarionBlobStorage(modelBuilder)` / `[GenerateElarionBlobStorage]` map the tables. Bundles its generator. | `Elarion.Blobs`, EF Core relational APIs, and Npgsql. |
| `Elarion.Blobs.AspNetCore` | Minimal direct blob-upload endpoint (`MapElarionBlobUploads`) producing pending blobs, for FilePond and plain `fetch`/`<form>` clients. | `Elarion.Abstractions`, `Elarion.Blobs`, and ASP.NET Core. |
| `Elarion.Blobs.Tus` | tus 1.0 resumable-upload transport (`MapElarionTus`) with an in-memory staging default; the recommended transport (Uppy `@uppy/tus`, `tus-js-client`). | `Elarion.Abstractions`, `Elarion.Blobs`, and ASP.NET Core; no storage dependency. |
| `Elarion.Blobs.Tus.PostgreSql` | Durable PostgreSQL staging store for tus uploads, plus the expired-session collector — `AddElarionTusPostgreSql<TDbContext>()`; `UseElarionTusStorage(modelBuilder)` / `[GenerateElarionTusStorage]` map the staging table. Bundles its generator. | `Elarion.Blobs.Tus`, `Elarion.Blobs`, and EF Core relational APIs. |
| `Elarion.Paging` | Keyset (cursor) and offset pagination execution (`[Keyset<T>]`, `IKeysetDefinition<T>`, opaque cursor codec, `SortMap`) plus the data-level authorization list filter (`[ResourceFilter<T>]`, `IQueryable.WhereAuthorized`). No DI registration — `IQueryable` extension methods. | `Microsoft.EntityFrameworkCore.Relational` and `Elarion.Abstractions`. |
| `Elarion.Messaging.InMemory` | Best-effort in-memory integration-event (Plane B) bus, commit-gated by the EF Core DbContext transaction — `AddElarionInMemoryIntegrationEventBus<TContext>()` (or `AddElarionInMemoryEventBus<TContext>()` for both planes). A non-durable sibling of the outbox; not needed with the transactional outbox. | EF Core relational APIs, `Elarion`, and `Elarion.Abstractions`. |
| `Elarion.Messaging.Outbox` | EF Core transactional outbox: a durable, at-least-once `IIntegrationEventBus` (Plane B) recording each event as an `OutboxMessage` row, delivered after commit by a polling worker — `AddElarionOutbox<TDbContext>(...)`; `UseElarionOutbox(modelBuilder)` / `[GenerateElarionOutbox]` map the table. Bundles its generator. | `Microsoft.EntityFrameworkCore.Relational` and `Elarion.Abstractions`. |
| `Elarion.JsonRpc` | Transport-neutral JSON-RPC dispatcher, envelopes, result/error types, telemetry, schema export, and the RPC method-map trigger. | `System.Text.Json` and Microsoft logging abstractions. |
| `Elarion.AspNetCore` | ASP.NET Core JSON-RPC endpoint mapping, batch execution, current-user middleware, and HTTP transport. | ASP.NET Core abstractions plus `Elarion.JsonRpc`. |
| `Elarion.AspNetCore.Identity` | Optional ASP.NET Core Identity **host wiring**: `AddElarionIdentity` (`AddIdentity` + EF stores), the Identity `ICurrentUser` claim mapping, and the transport-neutral authorizer. The web-free Identity *model* (`[GenerateElarionIdentity]`, `ApplyElarionIdentity`) lives in `Elarion.EntityFrameworkCore.Identity`. | `Microsoft.AspNetCore.Identity.EntityFrameworkCore` plus `Elarion.AspNetCore` (and the ASP.NET shared framework). |
| `Elarion.AspNetCore.Mcp` | Exposes JSON-RPC handlers as a Model Context Protocol (MCP) server for AI agents over Streamable HTTP. | `Elarion.JsonRpc` plus `ModelContextProtocol.AspNetCore`; the only Elarion package that references the MCP SDK. |
| `Elarion.AspNetCore.OpenApi` | Opt-in OpenAPI document generation for `[HttpEndpoint]` handlers (`AddElarionOpenApi()` + `app.MapOpenApi()`): wraps `Microsoft.AspNetCore.OpenApi` and adds the canonical-JSON wiring (schema gen with reflection off), module tags, normalized operation ids, and the `Idempotency-Key`/`x-elarion-idempotent` contract — REST parity with the JSON-RPC schema/client (ADR-0026). | `Elarion.AspNetCore` plus `Microsoft.AspNetCore.OpenApi` (pins `Microsoft.OpenApi` ≥ 2.7.5). |
| `Elarion.AspNetCore.SchemaGeneration` | MSBuild package and host-launching tool for exporting JSON-RPC schemas during build. | Private build-time reference from ASP.NET Core hosts. |
| `Elarion.EntityFrameworkCore` | Marker attributes for EF Core entity configuration and `DbSet` generation (`[EntityConfiguration]` on an `IEntityTypeConfiguration<T>` drives both the `DbSet` and its `Configure` application). Bundles the EF Core source generator (`DbSet` properties, entity configuration, keyset pagination) as an analyzer. | Marker-only at runtime; consumers provide EF Core. |
| `Elarion.EntityFrameworkCore.Identity` | The **web-free** ASP.NET Core Identity *model*: the `[GenerateElarionIdentity]` marker and `ApplyElarionIdentity` helper that compose a snake_case Identity model onto a plain `DbContext` (no `IdentityDbContext`). Lets a persistence/application layer model Identity without taking on the ASP.NET shared framework. Bundles its model generator as an analyzer. | `Microsoft.AspNetCore.Identity.EntityFrameworkCore` (EF Core + Identity stores) — **no** `FrameworkReference`. |
| `Elarion.Authorization.EntityFrameworkCore` | The data-level authorization **grants backend** (Leg B): a DB-native `ResourceGrant` table, `IResourceGrantStore`/`EfCoreResourceGrantStore`, and the grants-backed `IResourceAuthorizer` for `[RequireResource]` — `AddElarionResourceAuthorization<TDbContext>()`. Auth-provider-neutral (keys off `ICurrentUser` strings). Bundles its generator. | `Elarion.Abstractions` and EF Core relational APIs. |
| `Elarion.EntityFrameworkCore.UnitOfWork` | Framework-owned EF transaction / unit-of-work boundary: `EfUnitOfWork<TDbContext>` implements the EF-free `IUnitOfWork` seam over `BeginTransactionAsync` (PostgreSQL `SET LOCAL lock_timeout` + savepoints) — `AddElarionUnitOfWork<TDbContext>()`. The framework `TransactionDecorator` and idempotency both compose this boundary. | `Elarion.Abstractions` and EF Core relational APIs. |
| `Elarion.Idempotency.EntityFrameworkCore` | Durable EF Core idempotency store: `EfCoreIdempotencyStore<TDbContext>` claims a key with `INSERT … ON CONFLICT DO NOTHING` inside the caller's transaction (committed atomically with business writes; 409 on `lock_timeout` for an in-flight duplicate; success-only replay), plus the `IdempotencyKeyPurgeService` retention — `AddElarionIdempotencyEntityFrameworkCore<TDbContext>()`; `ApplyElarionIdempotencyKeys` / `[GenerateElarionIdempotencyKeys]` map the table. Targets PostgreSQL. | `Elarion`, `Elarion.EntityFrameworkCore.UnitOfWork`, and EF Core relational APIs. |
| `Elarion.Settings` | Runtime-changeable key/value settings with swappable abstractions (`ISettingsStore` sink + `ISettingsManager` consumer); ships the in-process default backend — `AddElarionSettings()`. | `Elarion.Abstractions` and `Microsoft.Extensions.{Configuration,DependencyInjection}.Abstractions`. |
| `Elarion.Settings.EntityFrameworkCore` | EF Core database backend for settings: `EfCoreSettingsStore<TDbContext>` with change-tracker-free immediate writes and optimistic concurrency — `AddElarionSettingsEntityFrameworkCore<TDbContext>()` (replaces the in-process store); `UseElarionSettings(modelBuilder)` / `[GenerateElarionSettings]` map the table. Bundles its generator. | `Elarion.Settings` and EF Core relational APIs. |
| `Elarion.Settings.Configuration` | Surfaces the `Global` settings scope as an `IConfiguration` provider with `IChangeToken` reload, so `IConfiguration`/`IOptionsMonitor` and `${...}` variable substitution pick up runtime changes — `builder.AddElarionSettingsConfiguration()`. | `Elarion.Settings` and `Microsoft.Extensions.{Configuration,Hosting.Abstractions,DependencyInjection.Abstractions,Logging.Abstractions}`. |
| `Elarion.Settings.PostgreSql` | Cross-instance change notification over PostgreSQL `LISTEN/NOTIFY`: a settings write on one node fires `IChangeToken` watchers on every node, and the EF store's writes are announced on their own connection so a transactional write is notified only on commit — `AddElarionPostgreSqlSettingsChanges(connectionString)` (or an `NpgsqlDataSource`). | `Elarion.Settings`, `Elarion.Settings.EntityFrameworkCore`, and `Npgsql`. |

## Architecture boundaries

The package split enforces clean dependencies:

- Core framework packages stay reusable and domain-neutral — they never reference a consuming app.
- `Elarion.Abstractions` does not depend on runtime integration packages.
- `Elarion` may depend on abstractions but avoids ASP.NET Core, EF Core, and transport specifics.
- `Elarion.Blobs` stays provider-neutral; `Elarion.Blobs.PostgreSql` owns PostgreSQL-specific dependencies and schema configuration. The upload **lifecycle** stays in the neutral core and is S3-free; the upload **protocol** is a source/transport at the HTTP layer only (`Elarion.Blobs.AspNetCore`, `Elarion.Blobs.Tus`, with durable tus staging in `Elarion.Blobs.Tus.PostgreSql`). The S3 wire protocol is a deliberate non-goal.
- `Elarion.JsonRpc` owns JSON-RPC runtime contracts and schema export without ASP.NET Core.
- `Elarion.AspNetCore` owns HTTP/JSON-RPC integration.
- `Elarion.EntityFrameworkCore.Identity` owns the web-free Identity model (`[GenerateElarionIdentity]`,
  `ApplyElarionIdentity`); `Elarion.AspNetCore.Identity` owns only the ASP.NET host wiring
  (`AddElarionIdentity`) and pulls the shared framework — the same split as
  `Elarion.EntityFrameworkCore` ↔ `Elarion.AspNetCore`. A data layer references the former without the web.
- `Elarion.AspNetCore.Mcp` is the only package that references the `ModelContextProtocol` SDK; the
  transport-neutral MCP tool model and dispatch shim live in `Elarion.JsonRpc`.
- `Elarion.AspNetCore.OpenApi` is the only package that references `Microsoft.AspNetCore.OpenApi`; the base
  HTTP transport stays OpenAPI-free, and the generator's tags/idempotent marker are inert without it.
- Validation follows the same seam/impl split: the `IRequestValidator` seam, `RequestValidationErrors`, and
  the `ValidationDecorator` live in `Elarion.Abstractions`; the `Microsoft.Extensions.Validation`-backed
  default lives in the opt-in `Elarion.Validation`. The constraint vocabulary is standard
  `System.ComponentModel.DataAnnotations` — no FluentValidation dependency anywhere in the framework.
- EF Core packages own only EF-specific markers and generation.
- Generators emit deterministic, inspectable code and fail with diagnostics for unsupported patterns,
  rather than scanning at runtime.

## Typical references

| Project | References |
| --- | --- |
| Application library | `Elarion` (bundles the generator); optionally `Elarion.Validation` (enforces the request DTOs' DataAnnotations and triggers the validation-resolver generation), `Elarion.Blobs`, `Elarion.EntityFrameworkCore` (bundles the EF Core generator). |
| Domain / entity library (if separate) | None — entities are plain classes with no marker. `[EntityConfiguration]` configurations live with the modules in the application library; the project holding the concrete `DbContext` references `Elarion.EntityFrameworkCore` so its generator reads the per-assembly configuration manifest and fills the context. |
| ASP.NET Core host | `Elarion` (for the generator), `Elarion.JsonRpc`, `Elarion.AspNetCore`; optionally `Elarion.Blobs.PostgreSql`, blob-upload transports (`Elarion.Blobs.Tus` + `Elarion.Blobs.Tus.PostgreSql`, or `Elarion.Blobs.AspNetCore`), `Elarion.AspNetCore.SchemaGeneration` (build tooling), `Elarion.AspNetCore.OpenApi` (OpenAPI for `[HttpEndpoint]`), and `Elarion.AspNetCore.Mcp` (MCP server). |
| Frontend | `@swimmesberger/elarion-jsonrpc-client-generator` + `zod`. |
