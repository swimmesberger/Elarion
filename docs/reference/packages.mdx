---
title: Packages
description: What each Elarion package contains, its expected dependencies, and when you need it.
---

Elarion is split into focused packages so you reference only what you use. Anything reusable moves
toward the `Elarion` family; application-specific domain concepts, database conventions, and
infrastructure stay in consuming projects.

## .NET packages

| Package | Purpose | Expected dependencies |
| --- | --- | --- |
| `Elarion.Abstractions` | Implementation-neutral **contracts only** — attributes, handler contracts, result types (including the `ElarionFile` binary file payload with its canonical base64 JSON envelope for the non-HTTP transports, ADR-0039), module metadata, scheduling contracts, the seam interfaces, the canonical JSON-serialization accessor (`IElarionJsonSerialization` / `ElarionJsonOptions` / `AddElarionJson` / `ConfigureElarionJson`), and source-generation triggers. Concrete behavior (the pipeline decorators, `HandlerTelemetry`) lives in `Elarion` core (ADR-0034). | Application-level abstractions without provider-specific dependencies. |
| `Elarion` | Runtime primitives: decorators, modules, current-user access, the default authorizer, the in-memory scheduler, the client-capability session bootstrap (`Elarion.Session` — `SessionHandler`/`AddElarionSession`, ADR-0030), and default-on [user-context trace/log enrichment](/docs/capabilities/telemetry#user--request-context) (`AddElarionUserContextEnrichment`, extensible via the `IHandlerContextEnricher` seam, ADR-0033), and **ordered streams** (`Elarion.Streams` — `StreamHub<T>`, ADR-0052): a hot, ordered, completable in-memory broadcast with atomic replay-then-live, a resumable replay ring, per-subscriber overflow strategy, and `Complete`/`Fail` — BCL-only (Channels + `IAsyncEnumerable<T>`), and **data-rate shaping helpers** (`Elarion.Buffering`, ADR-0055): `WriteBehindBuffer<T>` (batch loss-tolerant samples to the database by count/interval, bounded drop-oldest, single-flight, flush-on-dispose) and `KeyedConflater<TKey, TValue>` (latest-wins per key at a maximum publish rate, trailing emit so conflation never ends stale, idle keys retire) — BCL-only, no DI, `TimeProvider`-driven. The pipeline **decorators** (`Elarion.Pipeline` — observability (tracing + context enrichment merged, ADR-0059), authorization, feature-gate, validation, idempotency, resilience, caching, audit) and `HandlerTelemetry` live here (moved out of Abstractions, ADR-0034), calling seams that stay in `Elarion.Abstractions`; heavy provider implementations (caching, resilience, validation) are opt-in siblings (ADR-0017), so core depends only on `Microsoft.Extensions.*` abstractions. Bundles the Elarion source generator (handlers, services, modules, RPC/HTTP/MCP maps, validation resolvers, resilience policies, scheduled jobs, event consumers) as an analyzer. | `Elarion.Abstractions` and `Microsoft.Extensions.*` abstractions only; avoids ASP.NET Core, EF Core, and transport concerns. |
| `Elarion.Caching` | `HybridCache`-backed default `IHandlerCache` for handler result caching (`AddElarionHandlerCaching()`). **Required by `[Cacheable]`/`[CacheInvalidate]`** — the `HybridHandlerCache` was extracted out of `Elarion` core (ADR-0017). | `Elarion.Abstractions`; pulls `Microsoft.Extensions.Caching.Hybrid`. |
| `Elarion.Caching.PostgreSql` | The **recommended L2 distributed cache** for most apps: a PostgreSQL `UNLOGGED` table behind `HybridCache` (`AddElarionPostgreSqlHandlerCaching(connectionString)`), reusing your Postgres instead of operating Redis (`UseWAL = false` ⇒ an auto-created `UNLOGGED` table; ADR-0020). Redis stays the better L2 for very high-throughput, independently-scaled, or multi-region caches. | `Elarion.Caching`; pulls the official `Microsoft.Extensions.Caching.Postgres` (and Npgsql). |
| `Elarion.Resilience` | Microsoft/Polly-backed default `IResiliencePipelineRunner` (`AddElarionResilience()`). **Required by `[Resilient]` handlers and deferred scheduler retries** — the runner was extracted out of `Elarion` core, so `AddElarionScheduler` no longer auto-wires it (ADR-0017, breaking). | `Elarion`; pulls `Microsoft.Extensions.Resilience`. |
| `Elarion.Actors` | In-memory actor runtime ([Actors](/docs/concepts/actors), ADR-0042): plain classes marked `[Actor]` become keyed, mailbox-protected state machines with **source-generated typed facades** (`IActorSystem.Get<IOrderFulfillment>(orderId)`). Orleans-style single-threaded execution with opt-in `[Reentrant]` turn interleaving, virtual activation + idle passivation, per-activation DI scopes, per-call timeout as the deadlock backstop, and `Elarion.Actors` ActivitySource/Meter telemetry (caller span parents the actor span; exceptions keep their actor-side stack traces). Durable state via a declared `IActorState<TState>` constructor parameter — snapshot loaded before `OnActivateAsync`, explicit `WriteStateAsync`/`ClearStateAsync`, members mirroring Orleans' `IPersistentState<T>`, provider-neutral `IActorSnapshotStore` seam (ADR-0047). **Stream methods** (ADR-0052): a method returning `IAsyncEnumerable<T>` becomes a facade stream — the attach is a mailbox turn, enumeration runs off the mailbox (`ActorStreams.Defer`), the Orleans 7+ grain-interface shape; a live enumeration retains the activation against idle passivation (refCount lifetime), correctness passivations still complete hubs on deactivation (ELACT012 guards the shape). A **single-node** vertical concurrency primitive — clustering is a non-goal; outgrow it by moving to Orleans/Akka.NET/Proto.Actor (ADR-0025 seam philosophy). | `Elarion.Abstractions` and `Microsoft.Extensions.*` abstractions only; the `ActorRegistrationGenerator` ships in the `Elarion`-bundled analyzer. |
| `Elarion.Actors.PostgreSql` | PostgreSQL actor state snapshotting ([Actors](/docs/concepts/actors#durable-state-snapshots), ADR-0047) and the actor home lease (ADR-0048). Snapshots: the `elarion_actor_snapshots` table (`jsonb` payload, so actor state stays inspectable with plain SQL; `version` column as the ETag) behind `IActorSnapshotStore` + the `IActorStateReader` query companion — `AddElarionPostgreSqlActorSnapshots<TDbContext>()`; `UseElarionActorSnapshots(modelBuilder)` / `[GenerateElarionActorSnapshots]` map the table; change-tracker-free, version-guarded writes on a fresh DI scope per operation. Actor home: `AddElarionPostgreSqlActorHome<TDbContext>()` — sugar over the generic role lease (ADR-0049) electing the instance that runs `[Actor(SingleHomed = true)]` actors; gate the outbox on `IActorHomeLease.IsHeld` so event delivery follows the lease. Bundles its generator. | `Elarion.Actors`, `Elarion.Coordination.PostgreSql`, and EF Core relational APIs. |
| `Elarion.Coordination.PostgreSql` | PostgreSQL **role leases** ([ADR-0049](https://github.com/swimmesberger/Elarion/blob/main/docs/decisions/0049-role-leases.md)): heartbeat-renewed leader election for named roles — "exactly one instance is X right now" — as one conditional-upsert row per role on the app's database (`elarion_role_leases`; the row is the whole membership protocol, failover = expiry, immediate on graceful shutdown). `AddElarionPostgreSqlRoleLease<TDbContext>(o => o.RoleName = "…")` per role registers a keyed `IRoleLease` + its heartbeat; `UseElarionRoleLeases(modelBuilder)` / `[GenerateElarionRoleLeases]` map the table. Application clock only; `IsHeld` undershoots by a safety margin (the old holder stops before a new one can start); consumers gate per call/cycle. The row also carries the holder's **advertised address** (`AdvertisedAddress` / the `IInstanceAddressProvider` seam → `IRoleLease.CurrentHolderAddress`) — what the role-holder proxy (ADR-0050) forwards to. Deliberately **not** a distributed-lock API — coarse roles only. Backs the actor home (ADR-0048) and lease-gated outbox delivery. Bundles its generator. | `Elarion.Abstractions` (the `IRoleLease` contract) and EF Core relational APIs. |
| `Elarion.Validation` | `Microsoft.Extensions.Validation`-backed default `IRequestValidator` for [declarative request validation](/docs/concepts/validation) (`AddElarionValidation()`): DataAnnotations on request DTOs, enforced by the framework `ValidationDecorator` and exported to every schema surface (ADR-0027). **Required to enforce validation attributes at runtime** — without it they are schema-documented but unenforced (`ELVAL002`). The `IRequestValidator` seam stays in `Elarion.Abstractions`; the `ValidationDecorator` lives in `Elarion` core (ADR-0034). | `Elarion.Abstractions`; pulls `Microsoft.Extensions.Validation` (runtime base only — Microsoft's bundled validation generator is excluded). |
| `Elarion.Scheduling.EntityFrameworkCore` | Cross-instance scheduler coordination over EF Core/PostgreSQL: per-occurrence claim rows make each recurring job occurrence execute on exactly one node (exact slots for cron, one-interval dedupe windows for fixed-rate/fixed-delay), plus a claim-retention purge worker — `AddElarionSchedulerEntityFrameworkCore<TDbContext>()`; `UseElarionSchedulerClaims(modelBuilder)` / `[GenerateElarionSchedulerClaims]` map the table. Bundles its generator. Targets PostgreSQL. | `Elarion.Abstractions` and EF Core relational APIs. |
| `Elarion.FeatureFlags.OpenFeature` | Default `IFeatureFlagService` over OpenFeature's `IFeatureClient` for `[FeatureGate]`/`[FeatureVariant]`; maps `ICurrentUser` into the OpenFeature `EvaluationContext` (targeting key + `UserId`/`Groups`) off-HTTP, so gating works with any OpenFeature provider (`AddElarionOpenFeature()`). | `Elarion.Abstractions` and `OpenFeature`; bring your own OpenFeature provider via `services.AddOpenFeature(...)`. |
| `Elarion.FeatureFlags.FeatureManagement` | Batteries-included, config-driven flags: wires the OpenFeature Microsoft.FeatureManagement provider behind the OpenFeature-backed `IFeatureFlagService` (`AddElarionFeatureManagement(configuration)`). | `Elarion.FeatureFlags.OpenFeature`, `OpenFeature.Hosting`, and `OpenFeature.Contrib.Provider.FeatureManagement` (a prerelease — NU5104 suppressed deliberately, ADR-0016). |
| `Elarion.Blobs` | Provider-neutral blob storage contracts and DTOs, the upload lifecycle (`BlobLifecycleState`, `IBlobLifecycle`), prefix/delimiter listing (`ListAsync`/`ListContainersAsync`), and the staged (resumable) upload seam (`IStagedUploadStore`) with its in-memory default and the provider-neutral garbage collectors. | `Microsoft.Extensions.*` abstractions only; no provider-specific storage dependencies. |
| `Elarion.Blobs.PostgreSql` | PostgreSQL-backed blob storage with EF Core model configuration and Npgsql content I/O, the `IBlobLifecycle` implementation, and the durable `IStagedUploadStore` staging store with its expired-session collector — `AddElarionPostgreSqlBlobStore<TDbContext>()` / `AddElarionPostgreSqlStagedUploads<TDbContext>()`; `UseElarionBlobStorage(modelBuilder)` / `[GenerateElarionBlobStorage]` and `UseElarionStagedUploads(modelBuilder)` / `[GenerateElarionStagedUploads]` map the tables. Streaming reads clone the context's connection, so the context registration is the only connection wiring. Bundles its generator. | `Elarion.Blobs`, EF Core relational APIs, and Npgsql. |
| `Elarion.Blobs.AspNetCore` | Direct blob-transfer endpoints: the minimal upload/cancel surface (`MapElarionBlobUploads`, for FilePond and plain `fetch`/`<form>` clients) and the owner-scoped streaming download (`MapElarionBlobDownloads`) — the download leg of the staged-blob file tier (ADR-0039). | `Elarion.Abstractions`, `Elarion.Blobs`, and ASP.NET Core. |
| `Elarion.Blobs.Tus` | tus 1.0 resumable-upload transport (`MapElarionResumableBlobUploads`) — a pure protocol adapter over the `IStagedUploadStore` seam (in-memory staging by default); the recommended transport (Uppy `@uppy/tus`, `tus-js-client`). | `Elarion.Abstractions`, `Elarion.Blobs`, and ASP.NET Core; no storage dependency. |
| `Elarion.Blobs.Azure` | Azure Blob Storage backend: `AzureBlobStore` (`IBlobStore` + `IBlobLifecycle` over blob metadata) and the native `AzureStagedUploadStore` (append blobs with server-side append-position guards; completion is a server-side copy) — `AddElarionAzureBlobStore()` / `AddElarionAzureStagedUploads()`. | `Elarion.Blobs` and `Azure.Storage.Blobs`. |
| `Elarion.ClientEvents` | Client events (ADR-0043): after-commit facts projected to connected clients as at-most-once hints — `IClientEvent` topic catalog (`AddElarionClientEvents(events => events.AddTopic<T>("module.event"))`, per-topic subscribe requirements incl. `AllowAnyResource()` for routing-key resources), the canonical-JSON `IClientEventPublisher`, the producer-side subscription lifecycle (`ObserveSubscriptions<T>()`: per-subscriber greeting sink for initial values + linger-debounced interest transitions; `IClientEventInterest` pull check), and the in-process subscriber registry behind the replaceable `IClientEventBroadcaster` seam. | `Elarion.Abstractions` and DI/logging abstractions. |
| `Elarion.ClientEvents.PostgreSql` | Cross-node fan-out for client events over PostgreSQL `LISTEN/NOTIFY` (`AddElarionPostgreSqlClientEvents`): a publish on one node reaches the browsers connected to every node over the database the app already runs — dedicated listen connection per node, `elarion.connected` re-query signal after a reconnect gap, oversized (>8 KB) payloads fail loud at publish. | `Elarion.ClientEvents`, Npgsql, and hosting/logging abstractions. |
| `Elarion.ClientEvents.AspNetCore` | The Server-Sent Events transport for client events, built on the native `TypedResults.ServerSentEvents`: `MapElarionClientEvents(route)` streams a connection's authorized subscriptions with fail-closed subscribe-time authorization (unknown/denied topics read as 404) and named `elarion.connected`/`elarion.keepAlive` control events. | ASP.NET Core shared framework plus `Elarion.ClientEvents`. |
| `Elarion.Connections` | Bidirectional client-connection kernel (ADR-0053): the node-local `IClientConnectionRegistry` (lifecycle broker — observers run after the index mutation, failure-isolated), the transport-neutral codec seam `IClientConnectionProtocol` (complete inbound messages in receive order; outbound encoding codec-owned), `ClientConnectionTicket`, the kernel-wide options (`ElarionConnectionsOptions.DefaultInvokeTimeout` — every `InvokeAsync` is bounded by a 30 s default unless a per-call timeout is set; `null` = no default), and the client-events bridge (`ClientConnectionEventBridge` — the same subscribe pipeline and greeting as SSE, subscriptions die with their connection). Meter `Elarion.Connections`. | `Elarion.Abstractions`, `Elarion.ClientEvents`, and DI/logging abstractions. |
| `Elarion.Connections.AspNetCore` | WebSocket connection adapter, shaped for device gateways: `MapElarionConnectionSocket<THandler>` owns accept, message reassembly (size-capped), registry lifecycle, and teardown; the app's `WebSocketConnectionHandler` supplies the authenticator (HTTP-level or in-socket challenge/response) and the codec. `WebSocketClientConnection` adds serialized raw text/binary sends for proprietary codecs. | ASP.NET Core shared framework plus `Elarion.Connections`. |
| `Elarion.Connections.Tcp` | Raw TCP connection adapter for proprietary device protocols: hosted listener (`AddElarionTcpConnectionListener<THandler>`) and dial-out with jittered exponential reconnect (`AddElarionTcpConnectionDialer<THandler>`) over the same handler/codec seams; the adapter owns the byte-stream framing (`TcpMessageFramer` seam with `LengthPrefixedTcpFramer` and `DelimitedTcpFramer` built-ins — boundaries only; every inbound message reaches the codec's binary leg as a raw slice) so codecs receive complete messages. BCL sockets only. | `Elarion.Connections` and DI/hosting/logging abstractions. |
| `Elarion.Connections.Simulation` | Simulation and test utilities for connections — test suites and runnable dev/demo simulators alike: `SimulatedClientConnection` (in-memory sink double — capture sends, answer invokes, simulate close; registers with the real registry, proving the no-socket assumption), `AwaitableConnectionObserver` (deterministic connect/disconnect assertions), `TcpSimulatorClient` (framed client for simulated devices/peers over any `TcpMessageFramer`), and `InMemoryTcpLink`/`InMemoryDuplexStream` (the socket-free default: the full TCP connection lifecycle over an in-memory pair — reserve real sockets for integration-testing the adapters themselves). | `Elarion.Connections` and `Elarion.Connections.Tcp`. |
| `Elarion.Devices` | Device identity and provisioning ([ADR-0054](https://github.com/swimmesberger/Elarion/blob/main/docs/decisions/0054-device-identity-and-provisioning.md)) — the handshake companion to the connections stack: `IDevicePairingService` (issue a CSPRNG, human-typeable, TTL-bounded, single-use pairing code — the device id is pre-assigned at issue; redeem atomically claims it and mints the device's symmetric key; codes are stored SHA-256-hashed), the `IDeviceKeyStore`/`IPairingCodeStore` seams with explicit opt-in in-memory implementations, `HmacChallengeVerifier` (per-connection nonce + constant-time HMAC-SHA256 verification, shaped for `WebSocketConnectionHandler`/`TcpConnectionHandler` authenticators), and the `DevicePrincipal` factory (stable claim shape so `[RequirePermission]` and auditing work unchanged for device-initiated commands). `AddElarionDeviceIdentity()`. | DI abstractions only (BCL crypto). |
| `Elarion.Devices.EntityFrameworkCore` | Durable EF-backed device identity stores (ADR-0054): the `elarion_device_keys` and `elarion_device_pairing_codes` tables behind `IDeviceKeyStore`/`IPairingCodeStore` — `AddElarionDeviceIdentityEntityFrameworkCore<TDbContext>()` registers the whole chain; `UseElarionDeviceIdentity(modelBuilder)` / `[GenerateElarionDeviceIdentity]` map the tables. Change-tracker-free writes on a fresh DI scope per operation; the pairing-code claim is one `DELETE … RETURNING`, so concurrent redeems are single-winner across nodes. Bundles its generator. | `Elarion.Devices` and EF Core relational APIs. |
| `Elarion.Paging` | Keyset (cursor) and offset pagination execution (`[Keyset<T>]`, `IKeysetDefinition<T>`, opaque cursor codec, `SortMap`) plus the data-level authorization list filter (`[ResourceFilter<T>]`, `IQueryable.WhereAuthorized`). No DI registration — `IQueryable` extension methods. | `Microsoft.EntityFrameworkCore.Relational` and `Elarion.Abstractions`. |
| `Elarion.Messaging.InMemory` | Best-effort in-memory integration-event (Plane B) bus, commit-gated by the EF Core DbContext transaction — `AddElarionInMemoryIntegrationEventBus<TContext>()` (or `AddElarionInMemoryEventBus<TContext>()` for both planes). A non-durable sibling of the outbox; not needed with the transactional outbox. | EF Core relational APIs, `Elarion`, and `Elarion.Abstractions`. |
| `Elarion.Messaging.Outbox` | EF Core transactional outbox: a durable, at-least-once `IIntegrationEventBus` (Plane B) recording each event as an `OutboxMessage` row, delivered after commit by a polling worker — `AddElarionOutbox<TDbContext>(...)`; `UseElarionOutbox(modelBuilder)` / `[GenerateElarionOutbox]` map the table. Seeds each delivery scope with the message id (the default-on inbox's dedup key, `IEventContext.MessageId`) and wires the idempotency building blocks. Bundles its generator. | `Microsoft.EntityFrameworkCore.Relational`, `Elarion`, and `Elarion.Abstractions`. |
| `Elarion.Migrations` | The database-neutral SQL migration engine for EF-free (NativeAOT) hosts ([SQL migrations](/docs/capabilities/sql-migrations), ADR-0057/ADR-0060) — the AOT tier's schema tool; EF applications keep EF migrations. Owns the whole script model and policy: embedded `V{version}__`/`R__` discovery (timestamp versions recommended), SHA-256 checksums over BOM-stripped CRLF→LF-normalized content, out-of-order/repeatable planning, and the roll-forward `MigrationRunner` (each versioned script's history row `elarion_schema_history` commits in the script's own transaction — a failed transactional migration leaves no row, so there is no repair command; `-- elarion: no-transaction` for DDL a database forbids inside a transaction, with an explicit failed row + `IMigrationRunner.ResolveFailedAsync(version, Retry \| MarkApplied)` recovery; out-of-order default `Warn`; explicit `BaselineAsync` only; undo/placeholders/dialect *settings* rejected — a new engine is a new provider package). Drives a database through the `IMigrationDatabase`/`IMigrationSession` seam; contains **no** database driver. `AddElarionMigrationRunner(...)` is the shared registration (one-runner guard + migrate-before-ready hosted service). `IsAotCompatible`. | `Microsoft.Extensions.*` abstractions only. |
| `Elarion.Migrations.PostgreSql` | PostgreSQL provider for `Elarion.Migrations` (ADR-0057/ADR-0060): implements the seam with one dedicated connection under a **session-level** `pg_advisory_lock` (session scope survives crashes, never hangs `CREATE INDEX CONCURRENTLY`), the `to_regclass` existence check, and the dollar-quote-aware statement splitter for the `no-transaction` path. `AddElarionPostgreSqlMigrations(connectionString \| dataSource, o => o.AddScripts(assembly, prefix))` / `PostgreSqlMigrationRunner`. The multi-node PostgreSQL AOT tier. `IsAotCompatible`. | `Npgsql`, `Elarion.Migrations`, and `Microsoft.Extensions.*` abstractions. |
| `Elarion.Migrations.Sqlite` | SQLite provider for `Elarion.Migrations` (ADR-0060): the single-node / edge AOT tier (one file per node — the multi-node lock concern does not apply). Serializes concurrent runners with a per-file **in-process** lock (SQLite is single-process; `LockTimeout` bounds the wait, `busy_timeout` backstops cross-process); `locking_mode = EXCLUSIVE` is rejected (two connections deadlock promoting to exclusive). SQLite's full transactional DDL keeps the history-row-commits-with-its-script invariant identical, and the rarely-needed `no-transaction` path runs in autocommit. `CommandTimeout` does not apply (SQLite has no statement timeout). `AddElarionSqliteMigrations(connectionString, o => o.AddScripts(assembly, prefix))` / `SqliteMigrationRunner`. `IsAotCompatible`. | `Microsoft.Data.Sqlite` (patched `SQLitePCLRaw` native bundle), `Elarion.Migrations`, and `Microsoft.Extensions.*` abstractions. |
| `Elarion.Sql` | AOT-native SQL row mapping for EF-free hosts ([SQL mapping](/docs/capabilities/sql-mapping), ADR-0058) — the AOT tier's access half, completing the ADR-0057 schema half. `[SqlRecord]` generates an explicit sealed `{Type}SqlMapper : ISqlRowMapper<T>` (bundled generator): ordinals resolved by name once per result set, then synchronous typed `GetFieldValue<T>` row reads — no name lookups, no boxing, no per-column await — plus typed `BindParameters`, `TableName`/`Columns.*` constants plus the clause-free statement constants (`Insert`, `Select`, `Columns.AllAssignments` — pure column enumeration, never a predicate) for composing hand-written SQL, a static `Instance`, one per-assembly registration (`AddElarionSqlMappers`), and — since the row type is `partial` — a **self-mapping** partial (`ISqlRecord<T>`) so `db.QueryAsync<Order>($"{Order.Select} WHERE …")` resolves the mapper from the type (no mapper argument; `Order.Table`/`Order.Select` are typed trusted fragments, so no `:raw`). `[SqlColumn]`/`[SqlIgnore]`/`[SqlJson]` per property (JSON rides the canonical accessor, ADR-0023; `jsonb` under `[assembly: UseElarionSql(Provider = SqlProvider.Npgsql)]`). **No interception, ever**: an unmapped type (or a non-`partial` row, or a shadowed member) is a compile error (`ELSQL001`–`ELSQL011`), never a silent reflection fallback — the deliberate inversion of Dapper.AOT. Also injection-safe SQL interpolation: `new SqlStatement($"… WHERE id = {id} AND status IN {statuses}")` binds values as parameters at compile time (collections expand for `IN`; empty collections fail loud; `SqlStatement.Verbatim(col)` splices trusted identifiers — no `:raw`; fragments compose as values), `SqlWhere` accumulates optional filters (`WHERE (a) AND (b)` or nothing — the no-DSL dynamic-WHERE answer), and `DbDataSource`/`DbConnection` extensions cover reads (`QueryAsync`/`QueryFirstOrDefaultAsync`/`QuerySingleOrDefaultAsync`/`QueryUnbufferedAsync` streaming), scalars/non-query, and writes (`InsertAsync`/`InsertManyAsync`). Benchmarked at hand-written-ADO.NET parity in time and allocations (`*SqlMapping*` in `tests/Elarion.Benchmarks`). SQL stays hand-written — no LINQ, no change tracking, no statement generation. `IsAotCompatible`. | `Elarion.Abstractions` and `Microsoft.Extensions.DependencyInjection`/`Hosting` abstractions (the hosted service installs the canonical JSON accessor for self-mapping `[SqlJson]` rows). |
| `Elarion.JsonRpc` | Transport-neutral JSON-RPC dispatcher, envelopes, result/error types, telemetry, schema export, and the RPC method-map trigger. | `System.Text.Json` and Microsoft logging abstractions. |
| `Elarion.AspNetCore` | ASP.NET Core JSON-RPC endpoint mapping, batch execution, current-user middleware, and HTTP transport — including the file-download translation for `Result<ElarionFile>` handlers (`ElarionHttpResults.ToFileResult`, ADR-0039) and `[ModuleEndpoints]`, the attribute that declares a module's endpoint hooks from outside its assembly (ADR-0040). Also the **role-holder proxy** (ADR-0050): `UseElarionRoleHolderProxy(role, prefixes)` serves the listed prefixes on every instance by forwarding to the lease holder's advertised address before anything runs locally (one hop, loop-guarded, 503 + `Retry-After` during failover; installs nothing when no lease is registered), plus `AddElarionInstanceAddress()` for address auto-detection. And the **ordered-stream SSE endpoint** (ADR-0052): `MapElarionStream<T>(route, subscribe)` serves an `IAsyncEnumerable<StreamItem<T>>` as Server-Sent Events with `id:` = sequence, so `Last-Event-ID`/`?after=` reconnects resume from the producer's replay ring. | ASP.NET Core abstractions plus `Elarion.JsonRpc`. |
| `Elarion.AspNetCore.Identity` | Optional ASP.NET Core Identity **host wiring**: `AddElarionIdentity` (`AddIdentity` + EF stores), the Identity `ICurrentUser` claim mapping, and the transport-neutral authorizer. The web-free Identity *model* (`[GenerateElarionIdentity]`, `ApplyElarionIdentity`) lives in `Elarion.EntityFrameworkCore.Identity`. | `Microsoft.AspNetCore.Identity.EntityFrameworkCore` plus `Elarion.AspNetCore` (and the ASP.NET shared framework). |
| `Elarion.AspNetCore.Mcp` | Exposes JSON-RPC handlers as a Model Context Protocol (MCP) server for AI agents over Streamable HTTP. | `Elarion.JsonRpc` plus `ModelContextProtocol.AspNetCore`; the only Elarion package that references the MCP SDK. |
| `Elarion.AspNetCore.OpenApi` | Opt-in OpenAPI document generation for `[HttpEndpoint]` handlers (`AddElarionOpenApi()` + `app.MapOpenApi()`): wraps `Microsoft.AspNetCore.OpenApi` and adds the canonical-JSON wiring (schema gen with reflection off), module tags, normalized operation ids, and the `Idempotency-Key`/`x-elarion-idempotent` contract — REST parity with the JSON-RPC schema/client (ADR-0026). | `Elarion.AspNetCore` plus `Microsoft.AspNetCore.OpenApi` (pins `Microsoft.OpenApi` ≥ 2.7.5). |
| `Elarion.AspNetCore.SchemaGeneration` | MSBuild package and host-launching tool for exporting JSON-RPC schemas during build. | Private build-time reference from ASP.NET Core hosts. |
| `Elarion.EntityFrameworkCore` | Marker attributes for EF Core entity configuration and `DbSet` generation (`[EntityConfiguration]` on an `IEntityTypeConfiguration<T>` drives both the `DbSet` and its `Configure` application). Bundles the EF Core source generator (`DbSet` properties, entity configuration, keyset pagination) as an analyzer; the generated `ConfigureEntities` also declares the discovered domain entities' single-property `Guid` primary keys client-assigned (`ValueGeneratedNever`, explicit configuration wins — ADR-0038). | Marker-only at runtime; consumers provide EF Core. |
| `Elarion.EntityFrameworkCore.Identity` | The **web-free** ASP.NET Core Identity *model*: the `[GenerateElarionIdentity]` marker and `ApplyElarionIdentity` helper that compose a snake_case Identity model onto a plain `DbContext` (no `IdentityDbContext`). Lets a persistence/application layer model Identity without taking on the ASP.NET shared framework. Bundles its model generator as an analyzer. | `Microsoft.AspNetCore.Identity.EntityFrameworkCore` (EF Core + Identity stores) — **no** `FrameworkReference`. |
| `Elarion.Authorization.EntityFrameworkCore` | The data-level authorization **grants backend** (Leg B): a DB-native `ResourceGrant` table, `IResourceGrantStore`/`EfCoreResourceGrantStore`, and the grants-backed `IResourceAuthorizer` for `[RequireResource]` — `AddElarionResourceAuthorization<TDbContext>()`. Auth-provider-neutral (keys off `ICurrentUser` strings). Bundles its generator. | `Elarion.Abstractions` and EF Core relational APIs. |

## Architecture boundaries

The package split enforces clean dependencies:

- Core framework packages stay reusable and domain-neutral — they never reference a consuming app.
- `Elarion.Abstractions` does not depend on runtime integration packages.
- `Elarion` may depend on abstractions but avoids ASP.NET Core, EF Core, and transport specifics.
- `Elarion.Blobs` stays provider-neutral; `Elarion.Blobs.PostgreSql` and `Elarion.Blobs.Azure` own provider-specific dependencies and configuration. The upload **lifecycle** and the staged (resumable) upload **seam** stay in the neutral core and are S3-free; the upload **protocol** is an adapter at the HTTP layer only (`Elarion.Blobs.AspNetCore`, `Elarion.Blobs.Tus` — future resumable protocols such as the IETF draft reuse the same seam). The S3 wire protocol is a deliberate non-goal.
- `Elarion.JsonRpc` owns JSON-RPC runtime contracts and schema export without ASP.NET Core.
- `Elarion.AspNetCore` owns HTTP/JSON-RPC integration.
- `Elarion.EntityFrameworkCore.Identity` owns the web-free Identity model (`[GenerateElarionIdentity]`,
  `ApplyElarionIdentity`); `Elarion.AspNetCore.Identity` owns only the ASP.NET host wiring
  (`AddElarionIdentity`) and pulls the shared framework — the same split as
  `Elarion.EntityFrameworkCore` ↔ `Elarion.AspNetCore`. A data layer references the former without the web.
- `Elarion.AspNetCore.Mcp` is the only package that references the `ModelContextProtocol` SDK; the
  transport-neutral MCP tool model and dispatch shim live in `Elarion.JsonRpc`.
- `Elarion.AspNetCore.OpenApi` is the only package that references `Microsoft.AspNetCore.OpenApi`; the base
  HTTP transport stays OpenAPI-free, and the generator's tags/idempotent marker are inert without it.
- Validation follows the same seam/impl split: the `IRequestValidator` seam and `RequestValidationErrors` live
  in `Elarion.Abstractions`, the auto-attached `ValidationDecorator` in `Elarion` core (ADR-0034), and the
  `Microsoft.Extensions.Validation`-backed default in the opt-in `Elarion.Validation`. The constraint vocabulary is standard
  `System.ComponentModel.DataAnnotations` — no FluentValidation dependency anywhere in the framework.
- EF Core packages own only EF-specific markers and generation.
- Generators emit deterministic, inspectable code and fail with diagnostics for unsupported patterns,
  rather than scanning at runtime.
