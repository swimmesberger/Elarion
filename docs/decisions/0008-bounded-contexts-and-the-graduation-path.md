# ADR-0008: Bounded contexts and the graduation path

- Status: Proposed
- Date: 2026-06-25
- Related: [ADR-0001](0001-event-transaction-phase.md) (event planes / outbox),
  [ADR-0002](0002-cross-module-communication.md) (`[ModuleContract]`),
  [ADR-0007](0007-data-is-platform-module-as-plugin.md) (the data layer is application logic),
  [solution structure](../concepts/solution-structure.mdx).

## Context

ADR-0007 establishes that within one assembly the **data layer is shared application logic** that
**modules plug into as features**. That gives *feature* separation but explicitly **not** *data*
separation — every module reaches the whole database through one `DbContext`. Eventually an aggregate cluster earns genuine
isolation: its own schema, its own `DbContext`, no shared entities with the rest. That is a **bounded
context**, and it needs a different, stronger boundary than a feature module.

This ADR records the intended *direction* (hence `Proposed`): how bounded contexts fit a modular
monolith without coupling, how a system graduates into them incrementally, and an open question
ADR-0007 surfaced about plugins that introduce new data.

## Decision (direction)

1. **Design and packaging are independent axes.** *What* a boundary separates (its design) is
   orthogonal to *how* it is enforced and shipped (its packaging). Whether code lives in one assembly or
   several is a packaging/deployment concern and **must not drive the design** — the same data design can
   exist soft-enforced in one assembly or hard-enforced across assemblies.

   | Boundary | Separates (design) | Enforced by | Packaging |
   |---|---|---|---|
   | **Feature module** | **behavior** over a shared per-context data layer | the **analyzer** (`ELMOD002`) | one assembly |
   | **Mini bounded context** | **data** — own scoped `DbContext`, schema, and migration history | the **analyzer** (`ELMOD002` — entities placed inside the module are module-owned) | one assembly |
   | **Bounded context** | **data** — own `DbContext`, schema, and migration history | the **compiler** (separate assemblies; a context sees only another's `Contract`) | separate assembly |

   The first and last rows are the original poles: behavior split, soft, co-located vs. data split, hard,
   separated. The **mini bounded context** is the deliberate middle — a bounded context's *design* (owns
   its data, collaborates only by contract) with a feature module's *packaging* (one assembly). Because
   design and packaging are independent, it is a **sanctioned resting place, not a stepping stone**:
   separating data does not by itself mandate the compiler, so a team rests here as long as it likes and
   splits the assembly — swapping analyzer → compiler enforcement and gaining an independent deployment
   unit — only when *packaging* concerns (independent deployment, team autonomy) demand it, which may be
   never. The soft enforcement is the accepted trade-off for that freedom, and per-schema separation
   already gives partial physical protection while it stays soft.

2. **The fractal context layout.** A bounded context is itself a small Elarion app, with ADR-0007
   applied *inside* it (a shared per-context data layer + feature modules over it):

   ```
   MyApp.Api                      ← host: composes contexts, transport wiring, provider registration (UseNpgsql)
   MyApp.Context1.Contract        ← in-process surface other contexts depend on (create on demand)
   MyApp.Context1.Application      ← behavior + Domain + Persistence (config, DbContext, migrations — ADR-0007)
   MyApp.Context1.Infrastructure  ← context-specific intent-only adapters (its own mail / external clients)
   MyApp.Context2.Contract / .Application / .Infrastructure
   MyApp.SharedKernel             ← minimal cross-context primitives only (value objects — never entities)
   ```

3. **Cross-context collaboration is by reference, never by shared model.** No cross-context foreign
   keys, navigations, or joins — reference another context's data **by ID**. Compose two ways, both
   already in Elarion:
   - **Synchronous** → the other context's `Contract` (typed, in-process; the ADR-0002 `[ModuleContract]`
     graduated to an assembly boundary).
   - **Asynchronous** → integration events + the outbox (ADR-0001). Cross-context consistency is
     **eventual**.

4. **`Contract` ≠ public API.** `Context.Contract` is the surface *other contexts* call; the external
   API (HTTP/JSON-RPC) is generated by the host from the context's `[HttpEndpoint]`/`[RpcMethod]`
   handlers. They may share DTOs but serve different audiences and evolve on different clocks — keep
   them distinct so the public shape does not re-couple the contexts.

5. **A minimal cross-context `SharedKernel`.** Genuinely universal value types (`Money`, common
   primitives/enums) may live in one tiny, stable `SharedKernel` every context references. Keep it
   ruthlessly small — changes ripple to all contexts — and **never put a shared *entity* there**; an
   entity belongs to exactly one context.

6. **Graduate incrementally — four stages, no big-bang rewrite:**
   1. **One shared context** — `Domain` + `Persistence` + feature modules, one `DbContext` (where most
      apps start, per ADR-0007).
   2. **Scoped contexts in one assembly** — `[GenerateDbSets("ctx")]` / `[EntityConfiguration("ctx")]`
      partition entities into separate `DbContext`s within one assembly. A purely **logical /
      application-layer** split: the contexts share the **same database and schema** by default and the
      entity exemption still holds — it separates the *model in code*, not the data in the database. The
      cheapest precursor, and it already exists.
   3. **Mini bounded context** — a scope promoted to a real data owner: it takes its **own schema and its
      own migration history**, shares no entities, and collaborates only by `[ModuleContract]` +
      integration events — the **full bounded-context design**, still in one assembly and soft-enforced by
      the analyzer. Physical (schema) data separation enters **here**, not at the assembly split. This is a
      **legitimate long-term home**, not merely a migration step; what a stage-4 split adds — the compiler
      boundary and an independent deployment unit — is deferred until packaging demands it, possibly never.
   4. **Separate context assemblies** — the `Context.*` layout above; the same design, now **hard,
      compiler-enforced** and independently deployable. Because stage 3 already did the data work, this is
      mostly moving files into a `Context.Application` project and adding a `Contract` assembly — no schema
      migration, no untangling.

   The **by-ID reference discipline** of ADR-0007 is what makes each step cheap: if aggregates already
   reference each other by ID, 2 → 3 is "give the scope its own schema and turn the cross-aggregate
   lookups into contract calls," and 3 → 4 is a mechanical packaging move. Cross-aggregate navigations
   would have to be untangled first.

7. **Data ownership is just *where the entity lives* — the analyzer needs no special mode.** `ELMOD002` is
   a uniform **location** rule (shipped): a type inside an `[AppModule]` is module-internal, a type outside
   every module is shareable. So a module earns data ownership simply by placing its entities in its **own
   namespace** — they become module-internal, and a cross-module reference to them is flagged like any other
   internal type. Shared-kernel entities stay shareable because they live outside every module. There is no
   "entity exemption" to lift and no scope-aware analyzer to build; the only **direction, not yet shipped**
   part of a mini bounded context is the *physical* side — its own scoped `DbContext`, schema, and migration
   history (`[GenerateDbSets("ctx")]` + `[EntityConfiguration("ctx")]`).

8. **Earn each split.** A context is justified by a *real* data seam — an aggregate cluster with
   few/no shared entities, a distinct change cadence, a different owning team, or a planned extraction
   to a service. Splitting on speculation produces a distributed monolith: contract chatter and
   eventual-consistency cost with none of the autonomy benefit. Stay one shared context until a seam
   is undeniable — then adopt the mini-context design (stage 3) when the *data* seam appears and split
   the assembly (stage 4) when *deployment or team autonomy* demands it. The two triggers are independent:
   a mini bounded context is a sanctioned resting place, so staying there indefinitely is not a smell — an
   assembly split is earned by *packaging* needs, never by data separation alone.

## Open question — how a plugin owns new data

ADR-0007 notes that a feature introducing new data *extends the shared data layer*, so that data does not
"rip out cleanly" with the feature folder. How a module/plugin should own **new typed data** while
preserving the removability promise is unresolved. Candidate directions, to be decided:

1. **Typed schema extension inside modules** — a module declares its own typed entities/schema and
   contributes migrations. Hard to maintain: there is a single migration point, migration ordering
   and shared-model ownership get complicated, and removability is not automatic.
2. **An untyped `jsonb` "module data" table in the shared data layer** — module-specific data goes in a
   generic store keyed by module, and a framework cleanup hook decommissions it when the module is
   removed. Trades schema typing for clean, automatic removability.
3. **Require a bounded context when new typed data is needed** — introducing new typed data is the
   trigger to graduate to a context, which owns its own schema/migrations and is removable as a unit.

The **mini bounded context** reframes this question rather than sitting beside the options: it makes
option 3 cheap (graduating no longer requires an assembly split — a module owns new typed data *in place*)
and answers option 1's objections (each data-owning module has its **own `DbContext`**, so migration
ordering is independent and removability is "drop the module's schema," not a manual untangle). A feature
that needs new *typed* data then becomes a mini bounded context; untyped `jsonb` (option 2) stays the
choice only when the data is genuinely schemaless. Adopting this as the answer is the leaning, not yet the
decision.

This question is intentionally left open here; resolving it (or a future ADR) will pick a direction.

## Consequences

- The disciplines ADR-0007 recommends — shared data, configuration in `Persistence`, references by ID,
  collaboration via `[ModuleContract]` + outbox — are precisely the **on-ramp** to bounded contexts:
  they keep a future split a move-files-and-add-a-contract operation rather than a rewrite.
- **Packaging never gates design.** Because data ownership (schema, migrations, contracts) is separable
  from the assembly split, the bounded-context design is adopted when the *data seam* appears and the
  assembly is split when *deployment or team autonomy* demands it — two decisions on two clocks, neither
  blocking the other. A system may rest at the mini-context (data separated, one assembly) indefinitely;
  the assembly split is a sanctioned next step, not a mandatory one.
- **Status `Proposed`.** Stage 1, the stage-2 scopes mechanism, and the uniform location-based `ELMOD002`
  (so a module already earns data ownership by placing entities in its own namespace) exist today. The
  *physical* side of a mini bounded context (stage 3: per-scope schema + migration history), the
  separate-assembly layout (stage 4), the cross-context `SharedKernel` convention, and the new-data model
  are intended direction, not shipped behavior.
