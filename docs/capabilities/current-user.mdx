---
title: Current user
description: ICurrentUser gives handlers transport-neutral access to the authenticated user, without depending on HttpContext.
---

Handlers often need to know who is calling — for authorization, auditing, or
[user-scoped caching](/docs/capabilities/caching). Elarion exposes this through a transport-neutral
`ICurrentUser` abstraction so application code never depends on `HttpContext` or
`IHttpContextAccessor`.

This follows the framework's boundary rule: `Elarion` owns the abstraction; `Elarion.AspNetCore`
owns the HTTP integration that fills it in.

## The abstraction

```csharp
namespace Elarion.Abstractions.Identity;

public interface ICurrentUser {
    string UserId { get; }
    string? Email { get; }
    IReadOnlyList<string> Roles { get; }
    bool IsAuthenticated { get; }
    bool IsInRole(string role);
    bool HasClaim(string type, string value);
    IEnumerable<string> GetClaimValues(string type);
}
```

Inject it like any other service:

```csharp
[Service]
public sealed class OwnershipStamp(ICurrentUser user) {
    // Stamp the signed-in account onto a new row, so reads can be scoped per user.
    public string CurrentOwnerId => user.UserId;
}
```

## Wiring it in an ASP.NET Core host

The default implementation reads claims from the authenticated principal into a scoped snapshot.
Register it during service configuration and add the middleware after authentication:

```csharp
using Elarion.AspNetCore.Identity;   // AddElarionCurrentUser / UseElarionCurrentUser

builder.Services.AddElarionCurrentUser();

var app = builder.Build();

app.UseAuthentication();
app.UseElarionCurrentUser();   // after authentication, before endpoints
app.UseAuthorization();
```

`UseElarionCurrentUser()` copies claim values into a scoped `CurrentUserSnapshot` once per request.
Application handlers then read the snapshot, so they never touch `HttpContext` and stay testable —
in a unit test you simply register a fake `ICurrentUser`.

## Works across every transport

`ICurrentUser` resolves identically under plain HTTP endpoints, JSON-RPC (single and batch), and MCP —
no extra wiring. This matters because the JSON-RPC and MCP dispatchers run each call in a **fresh DI
child scope** for isolation, and a child scope does not inherit the request scope's scoped instances. So
the snapshot the request middleware initialized is not the one a JSON-RPC/MCP handler would otherwise
resolve.

Elarion bridges that gap explicitly (no `IHttpContextAccessor`, no `AsyncLocal`). When a request scope
exists (JSON-RPC, HTTP batch) the per-call scope **inherits** the snapshot the middleware already built —
the materialized claims are copied by reference, so nothing is re-parsed. MCP's per-call scope is rooted at
the session / application root (not the HTTP request scope — the middleware runs, but its snapshot lives in
a different scope tree the tool handler can't reach), so there the snapshot is built from the per-message
principal the transport stamps on each call (`RequestContext.User`), captured at the boundary in a
`DispatchScopeContext`. Authorization rides the same path, since the authorizer reads `ICurrentUser`.

### Inheriting your own scoped state

The simplest way to carry an existing request-scoped service into JSON-RPC / HTTP-batch handler scopes is
to make it `IScopeCopyable<T>` and register it with `AddDispatchScopeInherited<T>()` — the rail copies the
request-scope instance into each call scope (a no-op for MCP, which has no request scope):

```csharp
public sealed class TenantContext : IScopeCopyable<TenantContext> {
    public string? TenantId { get; set; }
    public void CopyFrom(TenantContext source) => TenantId = source.TenantId;
}

// services.AddScoped<TenantContext>();
// services.AddDispatchScopeInherited<TenantContext>();
```

For full control — e.g. seeding from the captured principal so it also works under MCP — implement
`IDispatchScopeInitializer` directly and register it with `TryAddEnumerable`. `inheritFrom` is the
originating request scope (or `null` when there is none, e.g. MCP):

```csharp
internal sealed class TenantScopeInitializer : IDispatchScopeInitializer {
    public void Initialize(IServiceProvider callScope, IServiceProvider? inheritFrom, DispatchScopeContext context) {
        if (context.TryGet<ClaimsPrincipal>(out var user) && user is not null) {
            callScope.GetRequiredService<TenantContext>().TenantId = user.FindFirstValue("tenant");
        }
    }
}

// services.TryAddEnumerable(
//     ServiceDescriptor.Singleton<IDispatchScopeInitializer, TenantScopeInitializer>());
```

## Mapping claims

By default the middleware maps standard claim types. Override them when your identity provider uses
different ones:

```csharp
builder.Services.AddElarionCurrentUser(options => {
    options.UserIdClaimType = "sub";
    options.EmailClaimType = "email";
    options.RoleClaimType = ClaimTypes.Role;
    options.DefaultRolesWhenAuthenticated = ["user"];
});
```

| Option | Default | Purpose |
| --- | --- | --- |
| `UserIdClaimType` | `"sub"` | Claim used for `UserId`. |
| `EmailClaimType` | `"email"` | Claim used for `Email`. |
| `RoleClaimType` | `ClaimTypes.Role` | Claim used to populate `Roles`. |
| `DefaultRolesWhenAuthenticated` | empty | Roles granted to any authenticated principal. |

## Relationship to caching

[`CurrentUser`-scoped caching](/docs/capabilities/caching) depends on this abstraction: the cache uses
`ICurrentUser.UserId` (hashed) to isolate entries per user. If you use `CurrentUser` scope, make sure
`AddElarionCurrentUser()` and `UseElarionCurrentUser()` are wired and the principal carries a user id
claim.
